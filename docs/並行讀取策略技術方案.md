# 行程並行讀取策略 - 技術方案

## 問題現況

### 目前架構的瓶頸
1. **Google Calendar API 配額限制**：
   - 免費配額：每分鐘 60 次讀取請求
   - 多人同時使用時容易超限
   
2. **Supabase 資料庫未充分利用**：
   - 已遷移 events 表，但主要用於同步，讀取時仍優先查 Calendar API
   - 資料庫沒有當成主要資料源

3. **生日行程邏輯不完整**：
   - 生日行程在 Calendar 同步時會產生
   - 但若同步未執行或資料庫未更新，前端看不到生日行程

---

## 並行讀取策略（優先採用方案）

### 核心概念

**資料庫作為主要資料源，Calendar API 作為更新來源**

```
┌────────────────────────────────────────────────────┐
│ 前端請求行程                                          │
└────────────────────────────────────────────────────┘
                         │
                         ▼
┌────────────────────────────────────────────────────┐
│ 後端邏輯：並行策略                                     │
│                                                    │
│  1. 立即從資料庫讀取（主要來源）                        │
│     ↓                                              │
│  2. 同時觸發 Calendar API 同步（背景執行）             │
│     └─ 有更新則寫入資料庫                            │
│     └─ 若達配額限制，跳過（不影響回應）                │
│                                                    │
│  3. 回傳資料庫資料給前端                              │
└────────────────────────────────────────────────────┘
```

### 實作步驟

#### 階段 1：調整讀取邏輯（優先）

**修改路由層**（`server/routes/calendar.js`）：

```javascript
// GET /api/calendar/month
router.get('/month', optionalLineUser, async (req, res) => {
  try {
    const year = parseInt(req.query.year);
    const month = parseInt(req.query.month);
    const type = req.query.type || null;

    // 1. 立即從資料庫讀取（主要來源）
    const events = await eventDbService.getEventsByMonth(year, month, type);

    // 2. 背景同步（不阻塞回應）
    syncCalendarInBackground(year, month).catch(err => {
      console.warn('背景同步失敗（不影響前端）:', err.message);
    });

    // 3. 立即回傳資料庫資料
    res.json({
      success: true,
      data: events,
      source: 'database',  // 可選：告訴前端資料來源
    });
  } catch (error) {
    res.status(500).json({
      success: false,
      message: error.message || '取得月份行程失敗',
    });
  }
});

// 背景同步函式（不阻塞主請求）
async function syncCalendarInBackground(year, month) {
  try {
    // 組時間範圍
    const timeMin = `${year}-${String(month).padStart(2, '0')}-01T00:00:00+08:00`;
    const lastDay = new Date(year, month, 0).getDate();
    const timeMax = `${year}-${String(month).padStart(2, '0')}-${String(lastDay).padStart(2, '0')}T23:59:59+08:00`;

    // 從 Calendar API 拿最新資料
    const calendarEvents = await calendarService.getGroupEvents(timeMin, timeMax);
    
    // 同步到資料庫
    await calendarSyncService.syncEventsToDb(calendarEvents, timeMin, timeMax);
    
    console.log(`✅ 背景同步完成: ${year}年${month}月`);
  } catch (error) {
    // 若達配額限制或其他錯誤，記錄但不拋出
    if (error.message.includes('quota') || error.message.includes('rate limit')) {
      console.warn(`⚠️ Calendar API 配額已達上限，使用快取資料`);
    } else {
      console.warn(`⚠️ 背景同步錯誤:`, error.message);
    }
  }
}
```

#### 階段 2：生日行程邏輯

**選項 A：資料庫直接產生生日行程（推薦）**

優點：不依賴 Calendar API，資料庫即可完整呈現

```javascript
// 在 eventDbService 新增函式
const ensureBirthdayEventsInDb = async (year, month) => {
  // 1. 從 members 取得所有有生日的成員
  const members = await memberDbService.getAllMembers();
  
  // 2. 篩選該月份的生日
  const birthdaysThisMonth = members.filter(m => {
    const parsed = parseBirthdayMonthDay(m.birthday);
    return parsed && parsed.month === month;
  });
  
  // 3. 對每個生日檢查資料庫是否已有，沒有則新增
  for (const member of birthdaysThisMonth) {
    const { day } = parseBirthdayMonthDay(member.birthday);
    const eventId = `birthday_${member.lineId}_${month}_${day}`;  // 固定 ID
    const title = `${member.name}生日`;
    
    // Upsert 到資料庫
    await db.query(`
      INSERT INTO events (id, title, start_at, end_at, all_day, type, is_birthday_event)
      VALUES ($1, $2, $3, $4, true, '活動', true)
      ON CONFLICT (id) DO NOTHING
    `, [
      eventId,
      title,
      `${year}-${String(month).padStart(2, '0')}-${String(day).padStart(2, '0')}T00:00:00+08:00`,
      `${year}-${String(month).padStart(2, '0')}-${String(day).padStart(2, '0')}T23:59:59+08:00`,
    ]);
  }
};
```

#### 階段 3：配額偵測與切換

```javascript
// 在 googleAuth.js 加入配額計數器
let apiCallCount = 0;
let apiCallResetTime = Date.now() + 60000;  // 每分鐘重設

const isQuotaExceeded = () => {
  if (Date.now() > apiCallResetTime) {
    apiCallCount = 0;
    apiCallResetTime = Date.now() + 60000;
  }
  return apiCallCount >= 50;  // 設為 50 保留緩衝
};

const incrementApiCall = () => {
  apiCallCount++;
};

// 修改 getCalendarClient 包裝 API 呼叫
const getCalendarClientSafe = async () => {
  if (isQuotaExceeded()) {
    throw new Error('quota_exceeded');
  }
  incrementApiCall();
  return await getCalendarClient();
};
```

## 優勢

1. **快速回應**：直接從資料庫讀取，不受 API 延遲影響
2. **配額管理**：API 達上限時自動切換，不中斷服務
3. **最終一致性**：背景同步確保資料最新，但不阻塞前端
4. **生日行程穩定**：資料庫直接產生，不依賴 Calendar

---

## 實作優先序

1. **立即**：調整路由為「先讀資料庫，背景同步」
2. **短期**：加入配額偵測，達上限時跳過同步
3. **中期**：生日行程邏輯改為資料庫直接產生

---

## 注意事項

- **資料一致性**：確保 Cron 定期同步（目前設為每天一次）
- **初始資料**：首次部署後需執行一次完整同步
- **錯誤處理**：背景同步失敗不影響前端回應
- **監控**：記錄 API 呼叫次數與配額狀況
