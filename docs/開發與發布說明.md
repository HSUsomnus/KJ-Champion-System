# LINE LIFF 行事曆 — 開發與發布說明

本文件說明**新版本**專案（React 前端 + Express API，部署至 Vercel）的**開發模式**（本機 + ngrok 測試）與**發布模式**（部署到 Vercel）。

---

## 一、專案結構概覽

```
Line_Liff/
├── api/
│   └── [[...path]].js       # Vercel Serverless 入口：/api/* 交給 Express
├── frontend/                 # React + Vite + Tailwind 前端
│   ├── src/
│   │   ├── api.js           # API 呼叫（/api/calendar/today 等）
│   │   ├── context/LiffContext.jsx  # LIFF 初始化與登入狀態
│   │   ├── components/      # Layout、BottomNav、EventCard、PageHeader
│   │   └── pages/           # CalendarPage、ListPage、MembersPage 等
│   ├── index.html
│   ├── package.json
│   ├── vite.config.js       # 本機開發時 proxy /api、/health → 8080
│   └── tailwind.config.js
├── server/                   # Express 後端（僅 API，不提供靜態檔）
│   ├── server.js            # 本機時 listen(8080)，Vercel 時只匯出 app
│   ├── config/              # googleAuth、lineConfig
│   ├── middleware/          # auth（LINE User ID 驗證）
│   ├── routes/              # calendar、member、profile、line
│   └── services/            # calendarService、sheetService、lineService 等
├── package.json             # 根目錄：dev、build、install:all
├── vercel.json              # 建置指令、輸出目錄、SPA rewrites
├── env.example              # 環境變數範例
└── docs/
    └── 開發與發布說明.md    # 本文件
```

- **本機開發**：前端 Vite 跑在 `http://localhost:5173`，後端 Express 跑在 `http://localhost:8080`；Vite 將 `/api`、`/health` 轉發到 8080。
- **Vercel 發布**：前端建置產物在 `frontend/dist`，由 Vercel CDN 提供；`/api/*` 由 `api/[[...path]].js` 轉交 Express 以 Serverless 執行。

## 二、開發模式：本機 + ngrok 測試

### 2.1 事前準備

- **Node.js**：建議 18 以上（專案 `engines` 為 `>=18.0.0`）。
- **環境變數**：複製 `env.example` 為 `.env`（專案根目錄與 `frontend/` 若需可各一份，本機主要用根目錄給 server 用），並填入實際值（Google、LINE、LIFF 等），見下方「環境變數一覽」。

### 2.2 安裝依賴

在專案根目錄執行：

```bash
npm run install:all
```

會執行根目錄的 `npm install` 以及 `frontend` 內的 `npm install`。

### 2.3 啟動本機開發

在專案根目錄執行：

```bash
npm run dev
```

- **後端**：`http://localhost:8080`（API：`/api/calendar/today`、`/api/line/liff-id`、`/api/members` 等）。
- **前端**：`http://localhost:5173`（Vite 會把 `/api`、`/health` 轉發到 8080）。

用瀏覽器開啟 `http://localhost:5173` 即可操作 React 介面；若尚未用 LINE 登入，會顯示「使用 LINE 登入」區塊。

### 2.4 本機跳過 LIFF（僅測 API）

若只想在本機測 API、不跑真實 LIFF，可在 `frontend` 目錄新增 `.env.local`：

```env
VITE_DEV_SKIP_LIFF=1
```

重啟 `npm run dev` 後，前端會使用模擬的 LINE User ID，無需在 LINE 內開啟即可呼叫需 `userId` 的 API。

### 2.5 使用 ngrok 測試 LINE / LIFF

要在 LINE App 內打開你的 LIFF，必須使用 **HTTPS 且可從外網連到** 的網址。本機可用 **ngrok** 對外暴露 5173（前端）。

1. **安裝 ngrok**  
   至 [ngrok 官網](https://ngrok.com/) 下載並安裝，或使用 `npm install -g ngrok`（若可用）。

2. **啟動專案**  
   先在本機執行 `npm run dev`，確認 5173、8080 皆正常。

3. **對外暴露前端埠**  
   另開一個終端機執行：

   ```bash
   ngrok http 5173
   ```

   記下 ngrok 給的 **HTTPS 網址**（例如 `https://abc123.ngrok-free.app`）。

4. **設定 LINE LIFF Endpoint URL**  
   - 前往 [LINE Developers](https://developers.line.biz/) → 你的 Channel → **LIFF** 分頁。  
   - 編輯你的 LIFF App，將 **Endpoint URL** 設為上一步的 **HTTPS 網址**（例如 `https://abc123.ngrok-free.app`），**不要**加結尾斜線。  
   - 儲存。

5. **在 LINE 內測試**  
   - 在 LINE 聊天室或選單中點擊進入你的 LIFF（或透過 Bot 發送的 LIFF 連結）。  
   - 應會開啟 ngrok 網址，載入 React 前端；前端會向「同源」的 `/api` 發請求。  
   - 此時 **API 仍是 localhost:8080**：若 LIFF 是從外網（ngrok）打開，瀏覽器同源是 ngrok 網址，`/api` 會請求到 **ngrok 的 5173**，再由 Vite dev server 的 proxy 轉到 **本機 8080**。  
   - 因此 **只暴露 5173 給 ngrok 即可**，不需再對 8080 開 ngrok；前後端都經由 5173 的 proxy 連到 8080。

6. **若 502 / 連線錯誤**  
   - 確認本機 `npm run dev` 兩支程式（server + frontend）都在跑。  
   - 確認 LIFF Endpoint URL 與 ngrok 的 HTTPS 網址完全一致。  
   - 若曾改過 LIFF 設定，可清除 LINE 快取或重開 LIFF 再試。

### 2.6 開發模式環境變數一覽（.env 根目錄）

| 變數名稱 | 說明 |
|----------|------|
| `GOOGLE_SERVICE_ACCOUNT_EMAIL` | Google 服務帳戶 Email |
| `GOOGLE_PRIVATE_KEY` | 服務帳戶私鑰（PEM，換行用 `\n`） |
| `GOOGLE_PROJECT_ID` | GCP 專案 ID |
| `GROUP_CALENDAR_ID` | 團體 Google 日曆 ID |
| `MEMBER_SHEET_ID` | 成員試算表 ID |
| `MEMBER_SHEET_NAME` | 試算表工作表名稱（如 `成員資料`） |
| `LIFF_ID` | LIFF App ID |
| `LINE_CHANNEL_ID` | Channel ID |
| `LINE_CHANNEL_SECRET` | Channel Secret |
| `LINE_CHANNEL_ACCESS_TOKEN` | Messaging API 長期 Token |
| `PORT` | 後端埠號（預設 8080） |
| `APP_URL` | 對外網址（ngrok 時可先填 ngrok 的 HTTPS 網址，部署後改為 Vercel 網址） |

## 三、發布模式：部署到 Vercel

### 3.1 事前準備

- **Git 倉庫**：程式碼已推送到 GitHub / GitLab / Bitbucket，且 Vercel 有權限存取該倉庫。
- **Vercel 帳號**：至 [vercel.com](https://vercel.com) 註冊（建議用 GitHub 登入）。

### 3.2 在 Vercel 建立專案並連線 Git

1. 登入 Vercel → **Add New… → Project**。  
2. 選擇 **Import Git Repository**，選取本專案所在的 repo。  
3. 在 **Configure Project** 頁面建議設定如下（若未自動帶入）：
   - **Framework Preset**：Other。  
   - **Root Directory**：`./`（專案根目錄）。  
   - **Build Command**：`npm run build`（會執行 `frontend` 的 `npm run build`，產出 `frontend/dist`）。  
   - **Output Directory**：`frontend/dist`。  
   - **Install Command**：預設 `npm install` 即可（Vercel 會先裝根目錄依賴，build 時會再進 frontend 執行 `npm install` 與 `npm run build`）。

若專案根目錄已有 `vercel.json`，且內含 `buildCommand`、`outputDirectory`，Vercel 會優先採用，無須在畫面上重複填。

### 3.3 設定環境變數（重要）

在 Vercel 專案 **Settings → Environment Variables** 中，將本機 `.env` 所用變數**依名稱與值**一併設定（可勾選 Production / Preview / Development）。

必填建議如下（與本機相同）：

| 變數名稱 | 說明 |
|----------|------|
| `NODE_ENV` | 填 `production` |
| `APP_URL` | **部署完成後**改為實際 Vercel 網址，例如 `https://你的專案.vercel.app`，**不要**結尾斜線 |
| `GOOGLE_SERVICE_ACCOUNT_EMAIL` | 同本機 |
| `GOOGLE_PRIVATE_KEY` | 同本機（PEM 整段，換行用 `\n`，勿多餘引號） |
| `GOOGLE_PROJECT_ID` | 同本機 |
| `GROUP_CALENDAR_ID` | 同本機 |
| `MEMBER_SHEET_ID` | 同本機 |
| `MEMBER_SHEET_NAME` | 同本機 |
| `LIFF_ID` | 同本機 |
| `LINE_CHANNEL_ID` | 同本機 |
| `LINE_CHANNEL_SECRET` | 同本機 |
| `LINE_CHANNEL_ACCESS_TOKEN` | 同本機 |

**注意：**

- `GOOGLE_PRIVATE_KEY`：若從 JSON 金鑰複製，只貼 **值** 的內容，換行用 `\n`；Vercel 環境變數不支援多行，需以 `\n` 表示換行。  
- 第一次部署可先將 `APP_URL` 設為 `https://你的專案.vercel.app`，部署完成後再確認是否與實際網址一致（含自訂網域時需一併更新）。

### 3.4 執行部署

- 若已連線 Git，推送至對應分支（例如 `main`）後，Vercel 會自動建置並部署。  
- 或於 **Deploy** 頁手動點 **Deploy** 觸發。  
- 建置完成後會得到一個網址，例如：`https://line-liff-calendar-xxxx.vercel.app`。

### 3.5 部署後必做兩件事

1. **更新 APP_URL**  
   到 **Settings → Environment Variables**，將 `APP_URL` 改為實際的 Vercel 網址（或自訂網域），儲存後可至 **Deployments** 對最新一次部署點 **Redeploy**，讓新變數生效。

2. **更新 LINE LIFF Endpoint URL**  
   - 開啟 [LINE Developers](https://developers.line.biz/) → 你的 Channel → **LIFF** 分頁 → 編輯 LIFF App。  
   - 將 **Endpoint URL** 改為你的 **Vercel 網址**（例如 `https://你的專案.vercel.app`），**不要**結尾斜線。  
   - 儲存。

完成後，從 LINE 內開啟 LIFF 應會載入 Vercel 上的 React 前端，並透過同源 `/api` 呼叫 Vercel Serverless 上的 Express API。

### 3.6 驗證部署是否成功

1. **健康檢查**  
   瀏覽器或 curl 開啟：  
   `https://你的專案.vercel.app/health`  
   應回傳 JSON：`{"status":"ok","timestamp":"..."}`。

2. **API**  
   `https://你的專案.vercel.app/api/line/liff-id`  
   應回傳 `{ "success": true, "data": { "liffId": "..." } }` 等。

3. **前端**  
   開啟 `https://你的專案.vercel.app/` 應看到行事曆介面；未登入時會顯示「使用 LINE 登入」。

4. **在 LINE 內測試**  
   從 LINE 聊天室或選單進入 LIFF，操作行事曆、成員、個人檔案等，確認與本機行為一致。

## 四、常見問題

### 4.1 部署成功但首頁 404 或白畫面

- 確認 **Output Directory** 為 `frontend/dist`，且 repo 中 `frontend` 可正常執行 `npm run build` 並產出 `dist`。  
- 確認 `vercel.json` 的 `rewrites` 沒有把根路徑誤指到其他位置；應有將非 `/api/*` 指到 `/index.html` 的規則（如 `"source": "/((?!api/).*)", "destination": "/index.html"`）。

### 4.2 取得團體行程失敗 / DECODER routines::unsupported（本機或 Vercel）

- 多為 **GOOGLE_PRIVATE_KEY** 格式錯誤：換行遺失、貼成單行、或多餘引號，導致 OpenSSL 解碼失敗。  
- **本機**（.env）建議寫法：  
  - 從 Google JSON 金鑰檔複製 `private_key` 的**值**（不含外層雙引號）。  
  - 在 .env 寫成**一行**，並把每個換行改成「反斜線 + n」：  
    `GOOGLE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\nMIIE...\n...\n-----END PRIVATE KEY-----\n"`  
  - 不要有多餘空格或引號包住整段。  
- **Vercel**：到 Settings → Environment Variables → 編輯 `GOOGLE_PRIVATE_KEY`，只貼 PEM 內容，換行用 `\n`，儲存後 Redeploy。  
- 程式已加強：若私鑰被貼成單行（BEGIN 後直接接 base64），會自動嘗試還原換行；若仍報錯，請依上述格式手動修正 .env。

### 4.3 LIFF 開不起來或白畫面

- 檢查 LINE Developers 的 **LIFF Endpoint URL** 是否為目前 Vercel 網址（含 `https://`，無結尾斜線）。  
- 用瀏覽器開發者工具查看 Console / Network 錯誤；若在手機 LINE 內，可參考專案內「手機 LINE 瀏覽器用 Eruda 看 Console」等文件。

### 4.4 本機用 ngrok 時 API 失敗

- 確認只對 **5173** 開 ngrok，且前端是透過 `https://xxx.ngrok-free.app` 開啟；不要對 8080 再開一個 ngrok。  
- 確認 `npm run dev` 兩支程式都在跑，且 Vite 的 proxy 設定為 `target: 'http://localhost:8080'`。

### 4.5 冷啟動較慢

- Vercel Serverless 閒置一段時間後會冷啟動，第一筆 API 請求可能較慢，屬正常；之後同實例會變快。

## 五、總結

| 模式 | 指令／方式 | 前端網址 | API |
|------|------------|----------|-----|
| **開發（本機）** | `npm run dev` | http://localhost:5173 | http://localhost:8080（經 Vite proxy） |
| **開發（ngrok）** | ngrok 暴露 5173，LIFF Endpoint 指到 ngrok HTTPS | https://xxx.ngrok-free.app | 仍經 5173 proxy → 本機 8080 |
| **發布（Vercel）** | Git push 或手動 Deploy | https://你的專案.vercel.app | https://你的專案.vercel.app/api/*（Serverless） |

依本說明完成「本機 + ngrok」與「Vercel 部署」後，即可在開發與正式環境中一致地使用新前端與 API。  
若需擴充列表頁、成員頁、個人檔案、新增／編輯行程等，可再對應接上 `server` 既有 API 與 `frontend/src/api.js` 的封裝。
