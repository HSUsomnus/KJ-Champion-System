# 邀請字卡除錯經驗總結

> 本篇記錄「邀請字卡」功能從無法顯示到正常運作，歷經約 6 小時反覆除錯的過程與教訓，供之後開發與 AI 協作時參考，避免重複鬼打牆。

---

## 一、現象與問題

- **現象**：點「邀請新夥伴」後，字卡沒有出來，或畫面上出現籠統錯誤（例如「無法取得邀請字卡」）。
- **環境**：LIFF 在 LINE App 內、透過 ngrok 連到本機伺服器。

---

## 二、曾走過的彎路（鬼打牆原因）

1. **過多條件判斷與 fallback**  
   用 `liff.isInClient()`、`liff.isApiAvailable()` 等做一堆分支，失敗又 fallback 成「文字分享」。結果：行為難預測，且失敗時看不到真正錯誤。

2. **錯誤訊息被蓋掉**  
   用自己寫的總結（例如「好友分享失敗」「無法取得邀請字卡」）當 alert，沒有把 **API 回傳的 message** 或 **LINE 回傳的 err.code / err.message** 顯示出來，導致無法對症下藥。

3. **直接開 ngrok 網址測 LIFF**  
   在 LINE 裡直接開 `https://xxx.ngrok-free.dev/members.html`，而不是用 `https://liff.line.me/{LIFF_ID}/members.html`。LIFF 未在正確情境下運行，分享與權限會異常。

4. **改後端沒重啟伺服器**  
   改了 `server/` 底下的程式碼（例如 `lineService.js`、`line.js`）但沒重啟 Node，前端一直打到舊邏輯，看起來像「改了沒用」。

5. **Flex 訊息格式過於複雜**  
   按鈕或區塊用了 LINE 不支援的屬性（例如某些 `color`、`height`），可能導致靜默失敗或只顯示 altText，沒有明顯報錯。

6. **LIFF 權限未開**  
   未在 LINE Developers Console 的 LIFF 設定裡勾選 `chat_message.write`，分享相關 API 無法正常運作。

---

## 三、最後採用的做法（什麼讓字卡出來）

| 項目 | 做法 |
|------|------|
| **流程** | 邀請流程**直接**呼叫 `liff.shareTargetPicker(messages)`，不做複雜分支，失敗就只顯示錯誤、不 fallback。 |
| **錯誤顯示** | 後端 catch 時回傳 `error.message`；前端顯示 API 的 `data.message` 或 `res.status` + 內容；`shareTargetPicker` 失敗時顯示 `err.code` / `err.message`。 |
| **後端錯誤** | 邀請 API 回傳 500 時，`message` 用實際的 `error.message`，不只用「產生邀請訊息失敗」。 |
| **前端防呆** | 用 `res.text()` 再 `JSON.parse`，避免伺服器回 HTML 時 `res.json()` 炸掉；非 JSON 時顯示狀態碼與內容前段。 |
| **Flex 簡化** | 提供極簡版字卡（`?minimal=1`）與按鈕只保留必要屬性（如 `style`），避免不支援屬性導致靜默失敗。 |
| **權限** | LIFF 在 Console 勾選 `chat_message.write`，權限變更後關閉 LIFF 再從 liff 連結重新進入。 |
| **測試方式** | 一律用 `https://liff.line.me/{LIFF_ID}/members.html`（可加 `?minimal=1`）在 LINE App 內開啟。 |

---

## 四、給之後的除錯與規則

- **改後端一定要重啟伺服器**：專案規則已加入「改後端要提醒重啟」。
- **錯誤只顯示實際內容**：不在前後加「失敗」「無法取得…」等總結，規則已寫入 `.cursorrules`。
- **LIFF 實機測試**：從 liff 連結進入、權限變更後重開 LIFF，已寫入專案規則。
- 若再遇到「改了沒反應」，可先檢查：  
  1）是否重啟過伺服器；  
  2）是否從 liff 連結進入；  
  3）LIFF 權限是否包含 `chat_message.write`；  
  4）畫面上的錯誤是否為「實際錯誤訊息」。

---

## 五、邀請字卡按鈕顏色（舊專案怎麼做的？）

### 結論：舊專案也是 LIFF + Flex，按鈕用 `color` 自訂色

- 舊專案（Python + HTML）**不是** Bot 發字卡，而是 **LIFF 網頁**裡用 `liff.shareTargetPicker([{ type: "flex", contents: flexContent }])` 分享 Flex 字卡，和現在 Node 專案**同一種流程**。
- 字卡 JSON 寫在 **前端**（member.html 的 `<script>` 裡），按鈕自訂顏色是這樣設的：
  - 每個 **button** 只設 **`color`**（例如 `"#4285F4"`、`"#F4B400"`、`"#00B900"`），**沒有**設 `backgroundColor`。
  - 有設 **`height: "sm"`**。
- 所以 **Flex Message 的 Button 支援用 `color` 當按鈕色**；官方文件沒寫清楚，但舊專案實作可行。

### 本專案目前做法（已對齊舊專案）

- 四顆按鈕皆為 **style: "primary"**，並只設 **color**（藍 `#4285F4`、橘 `#F4B400`、紫 `#9C27B0`、綠 `#00B900`），加上 **height: "sm"**。
- 已移除 `backgroundColor`，與舊專案一致，避免 API 解讀差異。
