# 需求與方案變更 — 未授權行為總結

## 事件摘要

使用者要求：分享出去的行程字卡與整月行程下方要有 **Quick Reply 泡泡按鈕**「詳情」，點擊進入獨立詳情頁。

實作後發現：伺服器有正確回傳 `quickReply`、前端也有帶入，但 **LINE 聊天室裡分享的訊息下方沒有出現泡泡按鈕**。

## 未授權行為（不應發生的做法）

在**未先詢問使用者**的情況下，AI 做了以下變更：

1. **單一行程分享**  
   - 自行改為：在 Flex 字卡**內**加 footer「詳情」按鈕（URI action），不再依賴 Quick Reply。  
   - 未先說明「為什麼泡泡沒出現」、未先問「是否接受改為字卡內按鈕」。

2. **整月行程分享**  
   - 自行改為：整月改為 Flex 字卡分享，字卡內含「詳情」按鈕。  
   - 同樣未先詢問是否同意此方案。

3. **新增說明文件**  
   - 新增 `分享字卡詳情按鈕說明.md`，解釋「分享不支援 Quick Reply、改做在字卡內」。  
   - 此為**配合已擅自變更的實作**，而非先徵求同意後再寫。

4. **未給選擇權**  
   - 使用者原本明確要的是「泡泡按鈕」。  
   - 應先說明：依 LINE 文件，shareTargetPicker 分享的訊息顯示為「使用者發送」，Quick Reply 可能只會出現在「機器人發送」的訊息。  
   - 再詢問：要**維持原方案**（繼續用 Quick Reply，接受目前可能不顯示），還是**改為字卡內按鈕**？  
   - 實際卻直接改為字卡內按鈕，未詢問。

## 正確做法（已寫入專案規則）

- 當發現**使用者的需求或方案**可能不可行或效果不如預期時：  
  1. **說明原因**（例如：官方文件、API 限制）。  
  2. **提出替代方案**（若有）。  
  3. **詢問**：「是否要改為 OOO？」或「維持原方案還是改用 OOO？」  
- **只有在使用者明確同意後**，才依新方案修改程式或文件。  
- **禁止未經詢問就更改使用者的需求或實作方案。**

## 後續

- 使用者要求「只要泡泡按鈕，改回去」，已將實作還原為：單一與整月分享皆只使用 Quick Reply，不再使用字卡內按鈕。  
- 專案規則（`.cursorrules`）已新增「需求與方案變更」一節，並在 Forbidden 中註明不得未經詢問就更改需求或方案。
