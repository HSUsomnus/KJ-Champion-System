# 舊有前端詳細說明（public/）

> 本文件針對專案中 **現行上線的前端**（`public/` 目錄）做技術面詳細說明，包含檔案職責、資料流、與後端／LIFF 的銜接方式，供維護或遷移時對照。

---

## 一、定位與範圍

- **「舊有前端」**：指專案根目錄下的 **`public/`**，由 Node.js 伺服器以 `express.static(publicPath)` 提供。
- **技術選擇**：原生 HTML + 單一 CSS 檔 + 多個獨立 JS 檔，**未使用** React、Vue 等框架。
- **執行環境**：主要於 **LINE App 內建瀏覽器**（LIFF）中開啟，亦可於一般瀏覽器用開發模式或 ngrok 測試。

---

## 二、技術棧與依賴

| 項目 | 說明 |
|------|------|
| **HTML** | 各頁獨立 `.html`，每頁自行載入所需 `<script>`、`<link>` |
| **CSS** | 單一 `public/css/style.css`，使用 CSS 變數、Flexbox、Grid |
| **JavaScript** | 原生 ES5+，無打包工具，每個頁面依需求引入多個 `.js` |
| **LINE LIFF** | 透過 `<script src="https://static.line-scdn.net/liff/edge/2/sdk.js">` 載入 LIFF SDK v2 |
| **除錯** | 網址帶 `?eruda=1` 時由 `eruda-loader.js` 動態載入 Eruda |

**無依賴**：無 npm 前端套件、無 Webpack/Vite 等打包、無 Babel。

---

## 三、目錄與檔案職責

### 3.1 目錄結構總覽

```
public/
├── css/style.css          # 全域樣式（變數、版面、元件、動畫）
├── images/logo.png        # 康九 logo（favicon、載入動畫）
├── js/                    # 各模組腳本，見下表
├── index.html             # 主頁：行事曆
├── list.html              # 行程列表
├── members.html           # 成員列表
├── member-detail.html     # 成員詳情（唯讀）
├── profile.html           # 個人資料／註冊
├── add-event.html         # 新增行程
├── event-detail.html      # 行程詳情／編輯
├── event-detail-standalone.html   # 分享用：單一行程詳情（可由 URL 參數指定 id）
├── month-events-standalone.html   # 分享用：整月行程
└── line-test-info.html    # 測試／除錯用
```

### 3.2 各 JS 檔案職責

| 檔案 | 職責 | 依賴 | 主要對外介面 |
|------|------|------|----------------|
| **liff.js** | 初始化 LIFF、取得 LIFF ID、登入狀態、觸發 `liffReady`、分享／開窗／登出 | LIFF SDK | `window.LIFF`（getUserId, getProfile, shareMessage, shareTargetPicker, login, isDevMode…） |
| **cacheService.js** | 版本檢查、快取清除、必要時觸發資料重載 | 無 | `window.cacheService`（checkAndUpdateData, clearCache） |
| **scroll-restore.js** | 頁面離開前儲存捲動位置，返回時還原 | 無 | 依 `sessionStorage` 與 `pageshow` 自動運作 |
| **share-dialog.js** | 取代 `alert`／`confirm` 的統一彈窗（單按鈕／雙按鈕） | 無 | `window.showAppAlert`, `window.showAppConfirm` |
| **calendar.js** | 行事曆主頁：月曆繪製、選取日期、當日行程載入與渲染、導向新增行程 | liff, cacheService, share-dialog, scroll-restore | 依 `liffReady` 與 DOM 事件驅動 |
| **list.js** | 行程列表：分頁（學員上課／活動／諮詢簽約）、依月份載入、分享整月、新增 | liff, cacheService, share-dialog, scroll-restore | 同上 |
| **members.js** | 成員列表：載入成員、渲染卡片、邀請新夥伴（分享字卡） | liff, cacheService, share-dialog, scroll-restore | 同上 |
| **member-detail.js** | 成員詳情：依 URL 參數載入單一成員、唯讀顯示 | liff, cacheService, share-dialog, scroll-restore | 同上 |
| **profile.js** | 個人資料：註冊狀態檢查、註冊表單、編輯／儲存、星等與課程紀錄 | liff, cacheService, share-dialog, scroll-restore | 同上 |
| **add-event.js** | 新增行程：表單驗證、日期時間、送出 API、可選分享 | liff, timePicker, datePicker, share-dialog, scroll-restore | 同上 |
| **event-detail.js** | 行程詳情：載入單一行程、顯示／編輯切換、更新／刪除、分享 | liff, timePicker, datePicker, share-dialog, scroll-restore | 同上 |
| **datePicker.js** | 滾輪式日期選擇器（年／月／日） | 無 | 依 DOM 與自訂 API 使用 |
| **timePicker.js** | 滾輪式時間選擇器（時／分） | 無 | 同上 |
| **eruda-loader.js** | 偵測 `?eruda=1` 並動態載入 Eruda | 無 | 無需手動呼叫 |

### 3.3 各 HTML 頁面載入的腳本順序（典型）

- **index.html**：eruda-loader → LIFF SDK → liff, scroll-restore, share-dialog, cacheService, calendar + 內嵌 entryGate
- **list.html**：eruda-loader → LIFF SDK → liff, scroll-restore, share-dialog, list
- **members.html**：eruda-loader → LIFF SDK → liff, scroll-restore, share-dialog, members
- **member-detail.html**：eruda-loader → LIFF SDK → liff, scroll-restore, share-dialog, member-detail
- **profile.html**：eruda-loader → LIFF SDK → liff, scroll-restore, share-dialog, profile
- **add-event.html**：eruda-loader → LIFF SDK → liff, scroll-restore, share-dialog, timePicker, datePicker, add-event
- **event-detail.html**：eruda-loader → LIFF SDK → liff, scroll-restore, share-dialog, timePicker, datePicker, event-detail

## 四、與後端 API 的對應

舊有前端透過 `fetch('/api/...')` 呼叫同一專案內 Express 後端，主要端點如下（供遷移時對照）。

| 用途 | 方法 | 路徑範例 | 備註 |
|------|------|----------|------|
| 取得 LIFF ID | GET | `/api/line/liff-id` | liff.js 初始化用 |
| 檢查成員註冊 | GET | `/api/members/check?userId=...` | 首頁 entryGate、profile 用 |
| 成員列表 | GET | `/api/members` | members.js |
| 單一成員 | GET | `/api/members/:id` 或依 query | member-detail.js |
| 個人資料 | GET/PUT | `/api/profile`（依 LINE userId） | profile.js |
| 註冊 | POST | `/api/profile/register` | profile.js |
| 當日行程 | GET | `/api/calendar/today?date=YYYY-MM-DD` | calendar.js |
| 月曆有無行程 | GET | `/api/calendar/events?year=&month=` | calendar.js |
| 依類型／月份行程 | GET | `/api/calendar/events?type=&year=&month=` | list.js |
| 新增行程 | POST | `/api/calendar/events` | add-event.js |
| 單一行程 | GET/PUT/DELETE | `/api/calendar/events/:id` | event-detail.js |
| 分享／邀請相關 | POST/GET | `/api/line/...`（依現有實作） | 分享字卡、邀請等 |

實際路徑與參數以 `server/routes/` 內定義為準；遷移到新前端時只需沿用同一 API 契約即可。

## 五、LIFF 生命週期與事件

1. **載入**：每個頁面載入 LIFF SDK 後，由 **liff.js** 執行 `liff.init({ liffId })`（liffId 來自 `/api/line/liff-id`）。
2. **開發模式**：`liff.js` 內 `APP_RUN_MODE = 'development'` 時不呼叫真實 LIFF，改為固定模擬 userId 並在下一 tick 觸發 `liffReady`。
3. **liffReady**：初始化完成後會 `window.dispatchEvent(new CustomEvent('liffReady', { detail: { userId, profile } }))`，各頁邏輯（calendar.js、list.js、profile.js…）依此載入資料或導向登入／註冊。
4. **未登入**：非開發模式下若未登入，liff.js 會動態插入 **liff-login-overlay**（或由 index 的 entryGate 顯示登入區），使用者點「使用 LINE 登入」後呼叫 `liff.login()`。
5. **分享**：在 LINE 內使用 `liff.shareTargetPicker()` 或 `liff.shareMessage()`；非 LINE 環境則 fallback 到 Web Share API 或複製到剪貼簿。

舊有前端**沒有**使用 React 等框架的 state，因此沒有「全域 store」；登入狀態與 userId 依賴 LIFF SDK 與 `liffReady` 事件在各頁各自處理。

## 六、關鍵共用機制

### 6.1 快取與版本（cacheService.js）

- 與後端版本比對，必要時清除前端快取並重新載入資料，避免看到過期版面或資料。
- 各頁在載入資料前可呼叫 `cacheService.checkAndUpdateData()`；標題列「重新整理」按鈕會呼叫 `cacheService.clearCache()` 後 `location.reload()`。

### 6.2 統一彈窗（share-dialog.js）

- `showAppAlert(message)`：單按鈕提示，取代 `alert`。
- `showAppConfirm(message, opts)`：雙按鈕確認，取代 `confirm`，回傳 Promise<boolean>。
- 樣式與分享確認框一致，避免原生彈窗破壞 LIFF 體驗。

### 6.3 捲動還原（scroll-restore.js）

- 離開頁面時將捲動位置存到 sessionStorage，返回時（含 bfcache 還原）還原捲動，避免使用者回到列表時從頭捲起。

### 6.4 底部導航與 active 狀態

- 各 HTML 自行撰寫 `<nav class="bottom-nav">`，依目前頁面為對應的 `<a>` 加上 `class="active"`，樣式由 `style.css` 的 `.nav-item.active` 控制。

## 七、獨立頁（standalone）用途與參數

- **event-detail-standalone.html**：供「分享單一行程」連結使用，可由 URL 參數傳入行程 id，頁面載入後只顯示該行程詳情，不依賴從主站導向過來的狀態。
- **month-events-standalone.html**：供「分享整月行程」使用，可由 URL 參數傳入年／月，顯示該月行程列表。

兩者皆為多頁式架構下對「單一連結可直連」的補充，若新前端改為 SPA，可改為同一 SPA 的 route（如 `/event/:id`、`/month/:year/:month`）並沿用相同 API。

---

## 八、伺服器如何提供舊有前端

- **靜態根目錄**：`server/server.js` 以 `express.static(publicPath)` 提供 `public/`，因此 `index.html`、`list.html`、`css/style.css`、`js/*.js`、`images/logo.png` 等皆可直接透過路徑存取。
- **根路徑**：`app.get('/', ...)` 回傳 `public/index.html`，故造訪 `/` 即為行事曆主頁。
- **favicon**：`/favicon.ico` 導向 `public/images/logo.png`，避免 404。
- **API**：所有 `/api/*` 由 Express 路由處理，與靜態檔分離。

遷移時若改為提供新前端（例如 React build 產物），可改為指向新前端的靜態目錄，並維持 `/api/*` 不變；若新前端為 SPA，需設定 fallback 將非 API 請求回傳單一 HTML（如 `index.html`）。

## 九、小結：舊前端的特點與注意點

| 面向 | 說明 |
|------|------|
| **優點** | 無建置步驟、部署單純、與現有 API 已打通、LIFF 流程已驗證 |
| **結構** | 多頁、每頁獨立 HTML + 多個 script，無元件化、無集中 state |
| **樣式** | 單一 CSS 檔、CSS 變數、類別命名與版面見 `docs/前端詳細敘述與風格指南.md` |
| **遷移時** | 需在新前端重現：liffReady 流程、同一組 API、分享／登入／快取與彈窗行為；standalone 可對應為 SPA 路由 |
