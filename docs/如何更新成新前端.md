# 如何更新成新前端

> 本文件說明如何將專案從 **舊有前端**（`public/` 多頁 HTML）更新為 **新前端**（`frontend-design/react` 的 React 單頁應用），包含選項、步驟、API／LIFF 銜接與部署注意事項。

---

## 一、現況對照

| 項目 | 舊有前端（public/） | 新前端（frontend-design/react） |
|------|---------------------|----------------------------------|
| **位置** | 專案根目錄 `public/` | `frontend-design/react/` |
| **技術** | 原生 HTML + 單一 CSS + 多個 JS | React 18 + Vite + Tailwind + React Router |
| **頁面** | 每頁獨立 `.html`，多頁導向 | 單一 SPA，路由：/、/list、/members、/profile、/add-event、/event/:id |
| **資料** | 已接後端 `/api/*`、LIFF | 目前為模擬資料（MOCK_*），尚未接 API／LIFF |
| **提供方式** | Express `express.static(public)` | 需建置後以 `dist/` 提供，或由 Vite dev 代理 |

---

## 二、更新策略選項

### 選項 A：完全替換（推薦用於正式切換）

- 以 React 建置產物（`dist/`）取代 `public/` 作為靜態來源。
- 後端不改 API，只改「靜態檔從哪裡來」。
- 優點：單一前端、維護簡單；缺點：需完整測試並規劃一次切換。

### 選項 B：並存後逐步切換

- 新前端部署在子路徑（例如 `/app`）或子網域，舊前端仍保留在 `/`。
- 透過功能開關或導向讓部分使用者先使用新前端，再全面切換。
- 優點：可漸進驗證；缺點：需維護兩套入口與路由設定。

### 選項 C：僅作預覽／內部使用

- 不替換現有對外服務，React 版僅供內部或設計預覽。
- 部署方式可為獨立靜態站或本機 `npm run dev`。

以下步驟以 **選項 A（完全替換）** 為主，其他選項可依需求調整路徑或部署目標。

---

## 三、前置：新前端需補齊的部分

在執行「替換」前，新前端需先與後端及 LIFF 對齊：

### 3.1 接上後端 API

- 將各頁的模擬資料（`MOCK_*`）改為呼叫現有 API。
- API 基底路徑建議使用相對路徑 `/api`（與現有一致），或透過環境變數設定（例如 `VITE_API_BASE`）。
- 對照 [舊有前端詳細說明.md](./舊有前端詳細說明.md) 第四節的 API 表，確保：
  - 行事曆：當日行程、月曆標記、依類型／月份列表、新增／更新／刪除行程
  - 成員：列表、單一成員、邀請
  - 個人：註冊檢查、取得／更新資料、註冊

### 3.2 接上 LINE LIFF

- 在 React 的 `index.html` 加入 LIFF SDK：  
  `<script src="https://static.line-scdn.net/liff/edge/2/sdk.js"></script>`
- 在 App 進入點（例如 `main.jsx` 或最外層 Layout）：
  - 先向後端取得 LIFF ID（`GET /api/line/liff-id`），再執行 `liff.init({ liffId })`。
  - 登入狀態可用 React state 或 Context 儲存（如 `userId`、`profile`）。
  - 初始化完成後觸發「liffReady」邏輯：檢查成員註冊、未登入則顯示登入 overlay、未註冊則導向註冊頁等，行為需與舊有前端的 entryGate 與各頁邏輯一致。
- 分享、開窗、登出等仍透過 LIFF SDK，可封裝成 hook 或 service 供元件使用。

### 3.3 環境變數

- 若新前端使用 Vite，可新增 `.env`／`.env.production`：
  - 例如 `VITE_API_BASE=/api`、`VITE_LIFF_ID=`（若改由前端直接帶 LIFF ID，否則仍建議由後端提供）。
- 與現有專案 `.env` 的 LINE 相關變數（LIFF ID 等）保持一致，避免重複設定。

### 3.4 共用行為對齊

- **快取／版本**：在適當時機（例如 App 掛載或進入主要列表前）呼叫與舊版相容的版本檢查或快取清除 API（若有），或沿用現有後端設計。
- **彈窗**：以 React 元件實作「單按鈕／雙按鈕」彈窗，取代 `alert`／`confirm`，行為與舊有 `share-dialog.js` 一致。
- **捲動還原**：若需「返回列表保留捲動位置」，可用 React state 或 sessionStorage 在離開前儲存、返回時還原。

## 四、完全替換的實作步驟（選項 A）

### 4.1 建置新前端

```bash
cd frontend-design/react
npm install
npm run build
```

- 產出在 `frontend-design/react/dist/`（預設為 Vite 建置結果）。

### 4.2 修改後端靜態來源

在 **server/server.js** 中：

1. 將原本指向 `public/` 的靜態目錄改為指向 React 的 `dist/`：

   ```js
   // 原本
   const publicPath = path.join(__dirname, '../public');

   // 改為（新前端）
   const publicPath = path.join(__dirname, '../frontend-design/react/dist');
   ```

2. 根路徑與 SPA fallback：
   - 保留 `app.get('/', ...)`，改為回傳 `dist/index.html`（或直接依下方 fallback 一次處理）。
   - 為支援 SPA 路由（如 `/list`、/members`、`/event/123`），在 **所有 API 與靜態路由之後**、404 之前，加入 fallback，將「非 API、非實際檔案」的 GET 請求都回傳 `dist/index.html`：

   ```js
   // 在 app.use('/api/...') 等之後、404 之前
   app.get('*', (req, res, next) => {
     if (req.path.startsWith('/api')) return next();
     res.sendFile(path.join(publicPath, 'index.html'));
   });
   ```

3. favicon：若新前端 `dist/` 內有 favicon（或 logo），可維持現有 `/favicon.ico` 邏輯改指到 `dist` 內路徑；或由 React 在 `index.html` 內設定 `link rel="icon"`，由靜態服務提供。

### 4.3 確認 API 與靜態路徑

- 新前端發出的請求應為相對路徑（如 `fetch('/api/calendar/today?date=...')`），與現有後端一致，無需改後端路由。
- 靜態資源（JS、CSS、圖檔）會由 `express.static(publicPath)` 從 `dist/` 提供，確保 Vite 建置產出的檔名與路徑正常（通常為 `assets/` 下帶 hash 的檔名）。

### 4.4 部署（以 Cloud Run 為例）

- 若目前部署會把 `public/` 一併打包，改為把 **建置好的 `dist/`** 一併打包進映像（或透過 build stage 在映像內執行 `npm run build`）。
- 環境變數（含 LINE、LIFF、API 網址等）與現有部署保持一致；新前端若用 `VITE_*`，需在 **建置時** 注入，執行期僅能使用建置時嵌入的值。

## 五、測試檢查清單

更新後建議至少確認：

- [ ] 根路徑 `/` 可開啟 React 主頁（行事曆）。
- [ ] 各路由可直連與重新整理：`/list`、`/members`、`/profile`、`/add-event`、`/event/:id`。
- [ ] LIFF 登入流程：未登入顯示登入區、登入後可取得 userId／profile。
- [ ] 成員註冊檢查：未註冊導向註冊／個人資料頁。
- [ ] 行事曆：當日行程、月曆、新增行程、進入詳情、編輯／刪除。
- [ ] 列表：分頁切換、依類型／月份載入、分享整月（若有）。
- [ ] 成員列表與詳情、邀請新夥伴（若有）。
- [ ] 個人資料：顯示、編輯、註冊送出。
- [ ] 分享行為：在 LINE 內可分享、非 LINE 有 fallback。
- [ ] 獨立分享連結：若保留 standalone 行為，可改為 SPA 路由並用相同 API 參數測試（如 `/event/123`、`/month/2024/1`）。

---

## 六、若需還原為舊有前端

- 將 **server/server.js** 的 `publicPath` 改回 `path.join(__dirname, '../public')`。
- 若有加 SPA fallback，移除或改回僅對 `/` 回傳 `public/index.html`。
- 重新部署即可回到多頁舊版前端。

建議在正式切換前保留舊有 `public/` 的備份或 Git 標記，以便快速還原。
