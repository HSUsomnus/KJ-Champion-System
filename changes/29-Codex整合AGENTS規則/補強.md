# Change 29 補強提案：AGENTS.md 與 CLAUDE.md 安全規則對齊

> 狀態：使用者已拍板「方案 R 瘦身修正版」（2026-07-18），Phase B1-B4 已實作，B5 上線待使用者確認。  
> 目的：補齊目前 `AGENTS.md` 相較 `CLAUDE.md` 遺漏的安全閘門與 skill 路由，同時避免把 Claude Code 專用操作原樣移植到 Codex CLI。  
> 實作差異：A4 未照原案「強制動手前完整讀取 SKILL.md」（與 spec 決策 B「不讓 Codex 自行跳讀」相衝突），改為「高風險操作停手交還指揮官 + skill 索引僅供參照」，更貼合 Codex 工兵定位。

## 一、檢核結論

目前 Change 29 的三個既定 gate 均通過，且沒有產品程式碼異動；但 `AGENTS.md` 標示「與 CLAUDE.md 對齊」，實際只移植部分鐵律。若 Codex CLI 確實只讀 root `AGENTS.md`、不讀 `CLAUDE.md`，下列缺口會形成獨立運作時的安全風險。

### A. 必須補強（上線前 blocker）

| 編號 | `CLAUDE.md` 規則 | 現況 | 風險 | 建議補法 |
|---|---|---|---|---|
| A1 | push main 前：now.md 同 commit、機密檢查、使用者明確確認 | `AGENTS.md` 未載明 | Codex 可能在沒有使用者確認或機密檢查下 push main | 在「Git 鐵律」新增完整條文 |
| A2 | prod / backup DB 寫入須回覆固定語句「確認操作正式 DB」 | 現行只寫「需使用者明確確認」 | 「可以」「好」等模糊回覆可能被視為授權 | 改成與 `CLAUDE.md` 完全一致的固定確認語句 |
| A3 | 需求或方案不可行時，先說明並詢問，不得自行改道 | 僅涵蓋架構判斷題，未涵蓋執行方案失敗 | Codex 可能在原方案失敗後自行換 API、套件或部署方式 | 在「邊界」新增不可自行改用替代方案 |
| A4 | 四種情境必須載入對應 skill | 只提到 workflow 路徑，沒有完整路由表 | UI、DB、部署工作可能漏讀專案專屬安全規則 | 新增「Skill 索引」，列出 workflow、uidesign、database、deploy-release，並明訂動手前先讀 |

### B. 建議補強（降低 VPS 新 session 誤操作）

| 編號 | 項目 | 判斷 | 建議補法 |
|---|---|---|---|
| B1 | 對話啟動時確認 branch、遠端 main、now.md | 對 Codex 同樣有用，但 Claude 原指令含 `git fetch`，會碰網路 | 新增 Codex 版啟動檢查：先看目前 branch、讀 `.claude/now.md`；需要依遠端 main 判斷時才 fetch，失敗須回報，不可假設本機 main 最新 |
| B2 | 已定案決策不得自行推翻 | Codex 是機械執行者，更需要知道架構邊界 | 不複製整張易過期表；要求實作前讀 `CLAUDE.md`「已定案決策」或 `.claude/now.md`，與既有 spec 衝突時停手 |
| B3 | 失敗兩次停手時的報告格式 | 現行已有核心內容，但「完整報錯」可能導致大量 context | 明確限定保留完整錯誤主體與必要 stack，刪除重複成功輸出；兼顧可診斷性與 context 控制 |

### C. 不建議直接移植

| `CLAUDE.md` 內容 | 不直接移植原因 |
|---|---|
| `/打包`、`/繼續`、`/compact` | Claude Code 專用命令，Codex CLI 不保證支援相同語意 |
| `/規劃`、`/實作`、`/診斷` 打卡 | Claude session 角色系統，不是 Codex CLI 的通用介面 |
| 「收尾員（Haiku）」 | 模型與子代理配置屬 Claude 工作流，Codex 不應假設可用 |
| 整份「已定案決策」逐字複製 | 會造成雙份規則日後漂移；應引用單一來源並設定衝突停手規則 |

## 二、建議修改範圍

### 方案 R（建議）：最小安全補強

只修改 root `AGENTS.md`，加入：

1. push main 三閘門。
2. prod / backup DB 固定確認語句。
3. 不可自行改用替代方案。
4. 四項 skill 強制路由表。
5. 簡化版 session 啟動檢查與已定案決策引用。

優點：修正安全缺口且維持 `AGENTS.md` 精簡；不複製大量容易過期的專案背景。  
代價：Codex 執行時仍需依引用讀 `.claude/` 下的規則檔。

### 方案 M：只補目前三個已確認問題

只加入 A1、A2、A4，不加入 A3、B1、B2、B3。

優點：改動最小。  
風險：仍沒有「方案不可行不得自行改道」與 session 起始狀態檢查，不能完整宣稱與 `CLAUDE.md` 安全規則對齊。

### 方案 F：完整複製 CLAUDE.md 規則

不建議。Claude 專用命令、角色與專案背景會使 `AGENTS.md` 膨脹，而且形成兩份需同步維護的規則來源。

## 三、方案 R 的預定文字結構

以下是內容結構，不是最終逐字稿；使用者確認後才寫入 `AGENTS.md`。

### 1. 新增「對話啟動」

- 確認目前 branch 與 HEAD。
- 讀 `.claude/now.md`。
- 需要比較 main 時，以 `origin/main` 為準；若無法 fetch，明確回報資料時間點，不假設本機 main 最新。

### 2. 補齊「Git／DB 鐵律」

- push main 前必須：更新 `.claude/now.md` 且與推送同 commit、完成機密檢查、列出推送清單並取得使用者明確確認。
- prod / backup DB 任何寫入，只有使用者明確回覆「確認操作正式 DB」才可執行。
- 需求或既定方案不可行時，先提出原因與替代選項，等待使用者決定，不自行切換方案。

### 3. 新增「Skill 索引」

| 情境 | 必讀 skill |
|---|---|
| 新功能、修改計畫、執行計畫、測試功能、修 bug、功能上線 | `.claude/skills/workflow/SKILL.md` |
| 新增或修改 `frontend/` UI 元件、樣式、顏色、彈窗、Tab、FAB、SidebarNav | `.claude/skills/uidesign/SKILL.md` |
| 編輯 `server/services/`、migration 或任何 DB 操作 | `.claude/skills/database/SKILL.md` |
| git push、merge、tag、上線、hotfix、部署設定 | `.claude/skills/deploy-release/SKILL.md` |

規則：符合情境時，動手前完整讀取對應 `SKILL.md`；找不到或讀不到就停手回報，不憑印象補規則。

### 4. 新增「既定決策與來源優先序」

- Codex 不做架構決策。
- 實作前以使用者本次指令、該 change 的 `spec.md`／`tasks.md`、`CLAUDE.md` 已定案決策與 `.claude/now.md` 為依據。
- 若來源互相衝突，列出衝突位置並詢問使用者，不自行選邊。
- 不修改 `spec.md`；需要調整規格時交回 Claude 與使用者。

## 四、檢核方案

### Gate 0：範圍與工作樹

目的：防止補強時混入 Claude 或使用者的其他改動。

```bash
git branch --show-current
git status --short
git diff --name-only <補強前HEAD>..HEAD
```

通過條件：

- branch 仍為 `m_b_AGENTS規則檔`。
- 實作補強的內容檔只修改 `AGENTS.md`；若依流程更新進度，另允許 `changes/29-Codex整合AGENTS規則/tasks.md`。
- 不加入現有未追蹤 `.agents/`、`.codex/`、`.kiro/`。
- 不修改 `server/`、`frontend/`、部署設定、migration 或 `.github/workflows/`。

### Gate 1：安全鐵律完整性

```bash
git grep -n "push main 前" -- AGENTS.md
git grep -n "機密檢查" -- AGENTS.md
git grep -n "使用者明確確認" -- AGENTS.md
git grep -n "確認操作正式 DB" -- AGENTS.md
git grep -n "不得自行改用\|不自行改用" -- AGENTS.md
```

通過條件：五項全部命中，而且 push main 條文同時包含 now.md、同 commit、機密檢查、使用者確認，不能拆成可被誤解的選擇項。

### Gate 2：Skill 路由完整性

```bash
git grep -n "workflow/SKILL.md" -- AGENTS.md
git grep -n "uidesign/SKILL.md" -- AGENTS.md
git grep -n "database/SKILL.md" -- AGENTS.md
git grep -n "deploy-release/SKILL.md" -- AGENTS.md
git grep -n "動手前.*讀\|完整讀取" -- AGENTS.md
```

通過條件：四個 skill 路徑全部命中，且明確要求「動手前完整讀取」，不是僅供參考的索引。

### Gate 3：Codex／Claude 邊界

人工檢查：

- 不出現要求 Codex 執行 `/打包`、`/繼續`、`/規劃`、`/實作`、`/診斷`。
- 不要求 Codex 使用 Haiku 收尾員或假設特定子代理模型存在。
- 保留「Codex 不做架構決策、不改 spec.md」。
- 規格衝突時必須停手詢問，不自行改規格。

### Gate 4：一致性與格式

```bash
git diff --check
git diff -- AGENTS.md
```

人工逐條比對：

- `AGENTS.md` 的 Git／DB 條文不得弱於 `CLAUDE.md`。
- 不得與 deploy-release、database、workflow、uidesign skill 發生相反指令。
- 使用繁體中文（台灣用語）；路徑、命令與固定確認語句精確。
- 沒有把「沙箱測試不可用」寫成一律跳過本機驗證；必須先確認失敗原因，再依 workflow/deploy 規則選擇部署驗證。

### Gate 5：回歸與交付前檢查

本 change 只有規則文件，不跑產品測試；以規則 gate 與 diff review 為主。

```bash
git diff --check main...HEAD
git diff --name-status main...HEAD
git log --oneline --decorate main..HEAD
git status --short
```

通過條件：

- 無 whitespace error。
- 仍為零產品程式碼異動。
- commit 歷史可追溯，沒有為整理歷史而 squash 已推遠端 commit。
- 執行 X1 前重新做 deploy-release 的機密檢查並取得使用者明確確認。
