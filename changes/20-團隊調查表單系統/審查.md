# Change 20 — 團隊調查表單系統可行性審查

> 審查日期：2026-07-18  
> 審查範圍：`spec.md`、`tasks.md`、`codex-delegation-spec.md`、`codex-delegation-tasks.md`，以及目前已落地的 `kj-survey-server/`、Survey 前端頁面與 `_worker.js`。  
> 審查性質：唯讀可行性審查。本文件不修改既有需求或技術決策；涉及方案選擇者，一律標示為待指揮官／使用者定案。

## 一、審查結論

整體產品架構與 Codex 委派方式具備可行性：Survey 後端獨立放在 `kj-survey-server/`，前端沿用既有 `frontend/`，四個後端工作包採 CX-1 → CX-2 → CX-3 → CX-4 序列執行，也符合共用 `routes/admin.js` 時避免平行衝突的原則。

但目前版本**不宜直接開始 CX-1～CX-4，也不可依現況上線**。審查發現：

- 2 項阻斷問題：後台認證可被偽造、draft token 規格與既有 DB schema 衝突。
- 3 項重要規格缺口：pending 名單污染未填統計、公開送出缺少後端驗證、部署網址與服務根目錄尚未確認。
- 1 項文件一致性問題：主 spec/tasks 與 delegation 文件對後台認證描述互相矛盾。

在下列問題定案並同步回正式規格前，Codex 工兵應停在委派前 gate，不得自行選擇替代方案。

## 二、阻斷問題

### B-1：後台授權可被冒用

**嚴重度：阻斷／安全性**

主規格與主 tasks 的既定方案是：

- LINE ID token 必須由後端驗簽，絕不信任前端傳值（`tasks.md` 20.17）。
- 後端簽發短效 JWT，放入 httpOnly、Secure、SameSite=Lax session cookie（20.19）。
- session 不落地 localStorage，關閉瀏覽器後失效。

目前實作與 delegation 文件則改成：

1. `SurveyAdmin.jsx` 將 LINE User ID 寫入 `localStorage.lineUserId`。
2. `surveyApi.js` 把該值放進 `X-Line-User-Id` header。
3. `requireAdminRole.js` 直接讀取 header，查詢 `members.role` 後放行。
4. delegation 文件要求所有 CX admin API 沿用此模式，並禁止重建 JWT／cookie／OAuth 驗證。

此流程只能證明「header 裡的 LINE ID 在 DB 中具有管理權限」，無法證明「呼叫者確實擁有該 LINE ID」。localStorage 與 HTTP header 都可由瀏覽器使用者自行修改；只要取得任一管理者 LINE User ID，即可能冒用管理者身分。

受影響範圍：

- CX-1：表單與所有填寫資料讀取。
- CX-2：完整成員與出席狀態資料。
- CX-3：CSV／Excel 全量匯出。
- CX-4：建立、修改及發佈表單。

即使 Jest 全綠，也只會證明 middleware 按傳入 header 運作，不能證明真實身分已被驗證。

**定案前不得執行：** CX-1～CX-4。

**待定案事項：** 由指揮官／使用者選擇可驗證呼叫者身分的正式認證方案，並先同步更新 `spec.md`、`tasks.md` 與兩份 delegation 文件。Codex 不自行決定沿用 cookie、改用後端簽發 bearer token，或採其他方案。

### B-2：draft token 流程與既有 DB schema 衝突

**嚴重度：阻斷／資料庫**

CX-4 規格要求：

- `POST /admin/forms` 建立 draft。
- `POST /admin/forms/:id/publish` 在表單尚無 token 時才產生隨機 token。

但已產生並標示已執行於 dev／prod／backup 的 migration 將欄位定義為：

```sql
token VARCHAR(64) NOT NULL UNIQUE
```

因此，若 `createDraftForm()` 按文字規格建立尚無 token 的 draft，DB 會因 `NOT NULL` constraint 拒絕寫入。

可行方向有兩種，但都屬規格決策：

1. 建立 draft 時就產生 token：不需修改 schema，但需改寫「發佈時才產生」的既定流程。
2. 將 draft 的 token 改為 nullable：符合目前 CX-4 文字，但需要新 migration，且涉及 dev／prod／backup DB schema 變更。

**定案前不得執行：** CX-4。

**待定案事項：** 指揮官／使用者選定 token 生命週期。若選擇 schema 變更，必須另外走 database skill 與正式 DB 雙閘門；本審查不授權 migration 或任何 DB 寫入。

## 三、重要規格缺口

### H-1：pending 名字會污染未填名冊

公開表單送出時，`name` 或 `recommender` 若不在既有名單，現有 `submitForm()` 會把新字串寫入 `survey_members`，狀態為 `pending`。

CX-2 又把「整份 `survey_members`」定義為應填名單。依此實作會把下列資料一起當成應填者：

- 姓名輸入錯字產生的 pending 記錄。
- 新輸入的推薦人名稱。
- 未經人工確認的陌生或惡意字串。

這會直接扭曲各推薦人分組的 total、filled 與進度比例。

**待定案事項：** 應填名單是否只包含 `status='confirmed'`。從現有欄位語意判斷，排除 pending 較符合用途，但這仍是需求／規格決策，Codex 不可自行採用。

### H-2：公開送出缺少伺服器端 schema 驗證

現有 `submitForm()` 會依 token 找表單並寫入 `answers`，但未完整驗證：

- `answers` 是否為 object。
- 必要欄位是否齊全。
- answer key 是否存在於 `form.fields`。
- `yesno` 是否為合法值。
- static searchable select（例如星等）是否符合 options。
- text 長度及可接受格式。

前端驗證不能取代後端驗證。若直接進入 Table 1、統計與匯出階段，異常資料可能造成欄位顯示錯誤、統計失真或匯出內容不可預期。

**建議：** 在繼續後台功能前，先由規劃層決定欄位必填規則與驗證錯誤格式，再新增一個後端 validation task 及對應 Jest。這會增加 task，應先更新 `spec.md`，不能只改 `tasks.md`。

### H-3：Survey 部署目標尚未完成校準

`frontend/public/_worker.js` 目前寫死：

- prod：`https://kj-survey.zeabur.app`
- dev：`https://kj-survey-dev.zeabur.app`

程式碼註解仍標示「待 Zeabur 服務建立後確認實際網址」。此外，repo 根目錄 `zbpack.json` 的 start command 是主系統 `node server/server.js`，因此新增 Survey Zeabur 服務時必須確認 Root Directory 設為 `kj-survey-server/`，否則可能部署成主系統後端。

**受影響 milestone：** 20.15、20.24、20.31、20.36、20.40、20.46、20.49。

**待確認事項：**

- dev／prod Survey 服務實際網址。
- 兩個服務的 Root Directory 與 start command。
- DB 內網環境變數是否指向正確的 dev／prod DB。

涉及 Zeabur 服務、`_worker.js` 或部署設定的修改均屬高風險操作，交由指揮官／使用者處理。

## 四、文件一致性問題

### D-1：認證規格存在兩套互斥的唯一事實來源

目前文件狀態如下：

| 文件 | 認證描述 |
|---|---|
| `spec.md` | LINE token 後端驗簽、短效 session、不落地 localStorage |
| `tasks.md` | 20.17～20.20 已勾選 JWT／cookie／middleware 測試完成 |
| `codex-delegation-spec.md` | 上述設計已移除，改信任 `X-Line-User-Id` |
| `codex-delegation-tasks.md` | 明令 CX 工作包禁止重建 cookie／JWT／OAuth |
| 現有程式碼 | 使用 localStorage + 可由客戶端指定的 header |

這不是單純註解過時，而是安全模型互斥。依專案決策來源規則，來源衝突時實作 session 必須停手，不能以 delegation 文件自行覆蓋主 spec。

**建議處理順序：**

1. 先由指揮官／使用者定案正式認證方案。
2. 先更新 `spec.md`。
3. 再同步 `tasks.md` 的完成狀態與待辦。
4. 最後更新兩份 delegation 文件與 CX prompts。

## 五、Codex 委派方案評估

排除上述阻斷後，委派架構本身可行，且邊界設計合理：

- 僅委派 `kj-survey-server/` 後端 API、service 與 Jest。
- `frontend/` UI、樣式與互動留給指揮官。
- `_worker.js`、DB、migration、部署與上線不交 Codex。
- CX-1～CX-4 共用 `routes/admin.js`，採序列而非平行。
- 每包只跑相關 Jest；每包完成後由指揮官 review diff，再決定是否進下一包。
- Codex 不 commit、不 push、不修改 spec，不自行換 API 或認證方案。

建議保留既定序列：

1. CX-1：後台表單與 submissions 讀取。
2. CX-2：未填名冊計算。
3. CX-3：CSV／Excel 匯出。
4. CX-4：表單建立器 API。

但應把「安全認證修正」設為 CX-1 前置 gate，把「draft token 定案」設為 CX-4 前置 gate。
