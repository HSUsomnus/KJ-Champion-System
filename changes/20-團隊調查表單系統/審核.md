# Change 20 — 團隊調查表單系統第二次可行性審核

> 審核日期：2026-07-18  
> 審核基準：`spec.md` v2 重寫版、`tasks.md` v2 重寫版，以及目前工作樹內的 Survey 前後端實作。  
> 本文件取代前次 `審查.md` 作為最新審核報告，但**不是需求或技術設計的唯一事實來源**；正式決策仍須先更新 `spec.md`，再同步 `tasks.md`。  
> 本次只做靜態審核，不修改產品程式碼、DB、部署設定、`spec.md` 或 `tasks.md`。

## 一、總結判定

v2 重寫方向明顯改善，前次審查的主要問題已在規格層完成以下處置：

- 不再信任可偽造的 `X-Line-User-Id`，改採 LINE 後端驗證後簽發短效 bearer JWT。
- draft 建立即產 token，與既有 `token NOT NULL UNIQUE` schema 相容，不需 migration。
- attendance 母數只取 `status='confirmed'`，排除 pending。
- 新增公開送出的後端 answers 驗證。
- 刪除分離的 delegation 文件，委派規則併回正式 spec/tasks。

但是，本版仍有 **3 項阻斷、5 項高優先缺口、4 項一般缺口**。目前判定為：

> **架構可行，但計畫尚不可直接執行，也不可上線。**

至少須先處理 B-1～B-3，並把 H-1～H-3 補入正式 spec/tasks，才適合開始 AUTH 與第一個 Codex 工作包。

## 三、阻斷問題

### B-1：OAuth `state` 的產生、完整性與一次性驗證未定義

**嚴重度：阻斷／認證安全**

`spec.md` 第三節定義 LINE OAuth 起始、callback、ID token verify 與 JWT 簽發，但沒有定義：

- 登入起始時如何產生高熵 `state`。
- callback 如何確認收到的 `state` 屬於同一次登入請求。
- `state` 如何防竄改、過期及重放。
- LINE 回傳 `error`、缺少 `code`、state 不符時的固定處理。

若只「比照主系統 `server/routes/auth.js`」，現有主系統是把資料 base64 編入 state；base64 不是簽章，也不提供一次性驗證。新 Survey 認證既然明確要求比主系統嚴，不能複製這個弱點。

可行做法不只一種，例如短效簽章 state、一次性 server-side state store，或安全 cookie 綁定；選擇會影響跨網域流程與部署，因此不能由實作者自行決定。

**必須先定案：** state 格式、簽章／保存方式、有效期、一次性消耗規則與錯誤回應。  
**必須補 tasks：** state 成功、缺少、竄改、過期、重放等測試。  
**定案前不得執行：** AUTH 20.8～20.15。

### B-2：前端登入 URL 與現有 Worker 代理路徑不一致

**嚴重度：阻斷／流程不可達**

目前 Cloudflare Worker 只代理：

```text
/survey-api/* → kj-survey-server，並移除 /survey-api 前綴
```

但 `tasks.md` 20.14 寫「未登入導 `/admin-auth/line-login`」。在前端 Pages 網域開這個路徑時，它不符合 `/survey-api/*`，會走 SPA fallback，而不是到 Survey 後端，因此 OAuth 起始 route 不會被呼叫。

正式規格必須明確選定並全程一致：

- 瀏覽器起始 URL 是 `/survey-api/admin-auth/line-login`，由 Worker proxy；或
- 使用 Survey Zeabur 後端的絕對 URL。

同時必須寫死 callback 的公開 URL、LINE Developers 登記的 callback URL，以及成功／失敗返回前端的 URL。不能只寫抽象的 `/admin-auth/...`。

**必須先定案：** dev/prod 各自的 login URL、callback URL、frontend return URL。  
**必須補 tasks：** Worker 代理 OAuth 起始與 callback/redirect 的 integration 驗證。  
**定案前不得執行：** AUTH 與 20.44 Worker 驗證。

### B-3：公開 `/members` 暴露完整成員與 pending 名單

**嚴重度：阻斷／個資與資料邊界**

目前 `GET /members` 完全公開，且 `listMembers()` 回傳：

```text
id, name, star_rank, recommender_name, status
```

這代表任何知道網站的人，不需持有有效表單 token，也不需登入，就能下載完整團隊姓名、星等、推薦關係以及 pending 名字。前台填表實際需要的主要是下拉選項，不必然需要公開上述全部欄位。

此外，pending 可能包含錯字、陌生人名稱或惡意輸入，不應再公開成下一位填表者的選項。

正式規格必須決定最小揭露範圍。較合理的方向是：

- 名單查詢綁定有效且 published 的 form token；
- 只回傳該表單欄位真正需要的最小資料；
- 一般下拉只使用 `status='confirmed'`；
- 是否顯示完整姓名、推薦關係，需由使用者確認隱私接受度。

這涉及產品資料邊界，Codex 不可自行縮欄位或改 endpoint。

**必須先定案：** 公開名單可揭露欄位、是否綁 form token、pending 是否永不公開。  
**必須補 tasks：** 未帶／無效 token 擋下、只回 confirmed、回傳欄位白名單測試。  
**定案前不得把現有 Section 1 的 `listMembers()` 判定為可保留。**

## 四、高優先缺口

### H-1：JWT 套件與環境範本未納入實作清單

目前 `kj-survey-server/package.json` 沒有 `jsonwebtoken`；`.env.example` 仍寫著「不需要 LINE_CHANNEL_ID/SECRET」，也沒有 `SESSION_SECRET`、OAuth callback URL 等 v2 必要設定。

Section 0 只要求使用者在 Zeabur 填環境變數，卻沒有 task 負責更新 repo 內：

- `package.json` 與 `package-lock.json`。
- `.env.example`。
- 啟動時必要環境變數檢查。

若直接執行 AUTH，會在 import JWT library 或讀取設定時失敗。

**建議補入 AUTH 前置 tasks：** 安裝並鎖定 JWT 套件、更新 `.env.example`、啟動時缺少必要 secret 應明確失敗、測試環境使用獨立測試 secret。

### H-2：`required` 與欄位 schema 尚未定義完整

CX-5 要驗證「每個 `required` 欄位」，但現有 migration 的 Phase 1 fields 沒有 `required` 屬性；現有 `SurveyFill.jsx` 則把所有欄位一律視為必填。Table 2 建立器也只寫 key/label/type/options，沒有 required 控制。

因此實作者無法確定：

- 舊 Phase 1 欄位缺少 `required` 時，預設是 true 還是 false。
- 新表單是否允許非必填欄位。
- `required` 是否要由建立器編輯並存入 JSON。
- `yesno` 合法值是否明確固定為現有前端使用的字串 `'yes'`／`'no'`。
- static select 的正式 schema 是 `options.values`，還是其他形式。

**建議在 spec 建立唯一欄位 schema：** 每種 type 的必要屬性、允許屬性、預設值、長度限制與合法 answer 型態。舊資料相容規則也要明寫，例如「缺少 required 視為 true」。

### H-3：公開送出不是 transaction，可能留下孤兒 pending 資料

現有 `submitForm()` 會先逐筆 INSERT pending member，最後才 INSERT submission，而且沒有 transaction。若 pending 寫入成功、submission 寫入失敗，DB 會留下沒有成功填表對應的 pending 名字。

CX-5 說「通過驗證才走既有寫入」，仍會保留這個資料一致性問題。

**建議補 tasks：** 整個「查 form → 建 pending members → 寫 submission」使用同一個 DB transaction；中途任一步失敗必須 rollback，並測試 rollback。這涉及 `config/db.js` client/transaction 用法，但不需要 schema 變更。

### H-4：表單建立器 API 缺少 server-side form schema 驗證

CX-4 目前只描述 create／patch／publish，沒有要求驗證：

- title 非空與長度。
- fields 必須是陣列。
- key 唯一且格式合法。
- type 只能是 `text`／`searchable_select`／`yesno`；Phase 2 `upload` 本次不可發布。
- options 必須符合該 type 的 schema。
- patch/publish 找不到 id 時回 404。
- published 表單是否仍可 patch。
- 空欄位表單能否 publish。

若不先定義，Table 2 可以建立前台無法 render、CX-5 無法驗證或匯出無法展開的表單。

**建議：** 在 spec 補 form definition validation 與狀態轉換規則，再納入 CX-4 Jest。

### H-5：匿名公開送出缺少濫用與重複提交邊界

`/forms/:token/submit` 免登入且 token 會被分享，現規格沒有 rate limit、request size、重複提交或垃圾資料處理規則。攻擊者可以大量建立 submissions 與 pending members，污染 attendance、匯出與 DB。

至少應定案：

- Express JSON body size 上限。
- 每 IP／token 的基本 rate limit 或其他節流方式。
- 同名重複提交是否允許；若允許，Table 1 與匯出如何呈現。
- 管理者是否需要刪除／排除垃圾 submission 的能力；若不做，需列為已知限制。

這可能改變 scope；應由指揮官／使用者判斷納入本 change 或明確接受風險，不由 Codex自行加套件。

## 五、一般缺口

### M-1：JWT 內角色在 4 小時內不會反映撤權

`requireAdminSession` 若只信 JWT payload 的 role，管理者被降權後，既有 token 仍可使用至到期。應明確接受「最長 4 小時撤權延遲」，或 middleware 每次再查 DB。這是安全性與 DB 負載的取捨，需定案。

### M-2：CSV 匯出缺少 formula injection 規則

CSV 儲存格若以 `=`, `+`, `-`, `@` 起頭，使用 Excel 開啟時可能被當成公式。answers 含使用者輸入，CX-3 除逗號／引號／換行逸出外，也應定義是否中和公式字首，並補測試。

### M-3：API 404／錯誤處理仍有未捕捉 async error

現有 `GET /forms/:token` 與 `GET /members` route 沒有 try/catch 或共用 async error middleware。DB query 拒絕時可能形成未處理的 rejected promise。Section 1「保留重驗」應把 DB failure response 納入驗證，而不只驗成功路徑。

### M-4：Section 0 gate 過度阻擋可離線完成的工作

tasks 宣稱 Section 0 未過就不下任何 `codex exec`，但 CX-5 的純函式與 Jest 不依賴 Zeabur URL、DB 內網或 `SESSION_SECRET`。這不會造成錯誤，但會讓部署人工前置阻塞所有可離線開發。

建議把 gate 拆成：

- 「實作 gate」：規格、安全與依賴已定案即可執行離線 code/Jest。
- 「milestone gate」：服務 URL、環境變數、DB 與部署完成後才跑 dev integration。

## 六、委派順序複核

v2 的工作包切分基本合理，但目前文字有輕微歧義：spec 先寫「AUTH → 之後 CX-1～CX-5序列」，後面又寫「CX-5 可最先委派」；tasks 則明確寫 AUTH → CX-5 → CX-1 → CX-2 → CX-3 → CX-4。

建議統一成下列依賴關係：

1. 規格修正：B-1～B-3、H-1～H-3 定案。
2. AUTH 基礎依賴與 `.env.example` 更新。
3. AUTH 實作與安全測試。
4. CX-5：answers validation + transaction。
5. CX-1：建立 admin route 與唯一 server 註冊點。
6. CX-2、CX-3、CX-4 依序追加。
7. 前端 UI 與各 Section milestone。

若希望提高效率，CX-5 可以在 AUTH 之前執行，因為它不碰 admin route；但必須先通過自己的欄位 schema、名單隱私與 transaction 規格 gate。是否採此順序應寫成唯一版本，不要同時保留兩種敘述。
