## 二、已達標部分

- 認證不再信任 `X-Line-User-Id`，改為 LINE verify 後簽發 bearer JWT。
- JWT 只放 `lineId`，middleware 每次重查 DB role，撤權即時生效。

- attendance 只以 confirmed 成員為母數。

- answers 後端驗證範圍、legacy `required` 預設、yesno 值及 text 長度已定義。
- pending member 與 submission 使用 transaction，失敗 rollback。
- CSV 公式字首中和、建立器基本狀態規則、async error handling 已納入。

- 分離 delegation 文件已移除，正式 spec/tasks 已成為單一事實來源。

## 三、阻斷性規格歧義

### B-1：OAuth callback URL 的 origin 與路徑語意衝突

spec 定義：

```text
瀏覽器登入入口：/survey-api/admin-auth/line-login
callback：{APP_URL}/survey-api/admin-auth/line-callback
後端實際 route：/admin-auth/line-callback
```

目前 Worker 只會在 Cloudflare Pages 網域把 `/survey-api/*` 去前綴後代理到 Survey 後端。如果 `APP_URL` 延續現有專案慣例，代表 Survey Zeabur 後端 origin，則：

```text
https://survey-backend/.../survey-api/admin-auth/line-callback
```

會直接打到 Survey Express；Express 沒有 `/survey-api/admin-auth/line-callback` route，因此 callback 404。只有 callback origin 是 Cloudflare Pages 時，`/survey-api` 才會被 Worker 去除。

必須在 spec 明確二選一，不能再用抽象 `{APP_URL}`：

1. LINE callback 登記為 `{FRONTEND_URL}/survey-api/admin-auth/line-callback`，經 Worker proxy；或
2. LINE callback 登記為 `{SURVEY_BACKEND_URL}/admin-auth/line-callback`，直接進後端。

並應分別列出 dev/prod 的完整範例 URL，以及 code 換 token 時使用的同一個 `redirect_uri`。定案後同步修改 20.0b、20.0c、20.16、20.17。

### B-2：OAuth state「一次性消耗」仍保留多個互斥方案

spec 寫：

> server 端短期記憶體／表記錄已消耗 nonce，或以簽章+exp+單次比對達成。

這仍不是唯一可實作方案：

- 記憶體 store：服務重啟會遺失，多 instance 不共享。
- DB table：需要 schema/migration，會觸發 database skill 與正式 DB 閘門。
- 只有 HMAC + exp：本身無法辨識同一 state 已使用過，不能實現重放阻擋；仍需可變狀態或與瀏覽器端一次性資料綁定。

目前 tasks 又要求測「重放」，所以實作者必須自行選 nonce store 才能完成。

必須定案：

- nonce 存放位置。
- TTL 清理方式。
- 多 instance／重啟行為是否接受。
- consume 是否需要原子操作。

若本服務確定單 instance，可明定使用 process memory `Map`、10 分鐘 TTL、callback 驗證成功前原子 consume，並接受重啟後未完成登入失效；若不能接受，需由指揮官另定共享 store。Codex 不可自行選擇。

### B-3：rate limit 沒有可實作的精確政策

目前只定義「每 IP/token 基本 rate limit」，但缺少：

- `windowMs`。
- `max`。
- key 的精確組成與 token 正規化。
- 超限 status/body。
- Zeabur／Cloudflare proxy 下 Express `trust proxy` 設定。
- 無效 token 是否也計入、如何避免用任意 token 逃避限制。

`express-rate-limit` 預設只按 IP；若未正確設定 proxy，所有請求可能被視為同一個代理 IP。若 key 直接含任意 URL token，也可能被輪換假 token 規避。

必須先寫死一組可驗收政策，例如「已解析的 client IP + 有效 form id、15 分鐘最多 N 次、429 固定 JSON」，並補 unit/integration test。數值與信任哪一層 proxy 屬部署／產品決策，不應交給實作者猜。

### B-4：tasks 排序與 workflow 的「下一個未完成 task」規則衝突

`tasks.md` 在已完成 A.1 後，下一個未完成項目是 Section 0 的 20.0a；但 Section 0 被定義為 milestone gate，不應阻擋離線實作。專案 workflow 又要求實作 session 找「下一個未完成且前置已完成的 task」，不跳躍。

此外，文件宣告的唯一順序從 AUTH 開始，但檔案順序先列 Section 1 與 Section 2，再列 Section 3 AUTH。不同執行者可能分別依「文件順序」或「執行前提醒」行動。

必須調整 tasks 結構，使機械執行不靠解讀例外：

- 把 milestone gate 移到 AUTH/PUB/CX 實作之後、第一個 dev milestone 之前；或
- 明確標示 Section 0 為非阻塞外部 dependency，不使用一般 `[ ]` task；或
- 按唯一執行順序實際重排 Sections：實作依賴 → AUTH → PUB → CX → milestone。

在 tasks 未重排前，不建議直接說「執行計畫」。

## 四、需要補齊的實作規格

### H-1：表單驗證仍缺精確數值與格式

H-4 已列驗證種類，但仍未定義：

- title 最大長度。
- field key 的 regex 與最大長度。
- label 最大長度。
- fields 最大數量。
- static `options.values` 最大項目數、單項長度、是否去重／允許空字串。
- `patchForm` 收到空 object、未知欄位或 `null` 的回應。
- validation fail 的 status 與統一 body shape。

這些值會直接形成 Jest 預期，應在 spec 寫死，否則 CX-4 仍需自行拍板。

### H-2：OAuth state 的 `returnMeta` 未定義白名單

state payload 包含 `returnMeta`，但沒有定義允許欄位與驗證方式。若其中含可控 return URL，可能重新引入 open redirect。此 change 實際只需固定回 `/admin`，最安全的做法是 state 不接受使用者提供的任意 origin／URL；成功與失敗目的地由後端環境變數組合固定路徑。

應明定 state payload 的完整 schema，並測試未知／惡意 return URL 不會影響 redirect。

### H-3：transaction 的 service API 邊界尚未落到 task

20.5 指定 Codex 實作 transaction，但現有 `config/db.js` 只暴露 pool/query 模式。task 應明確要求：

- `pool.connect()` 取得 client。
- 所有 transaction 內 query 必須使用同一 client。
- `finally` 一律 `client.release()`。
- BEGIN 後任何 error 都 ROLLBACK，且保留原 error 主體。
- `getPublishedFormByToken`、`listMembers` 是否需要接受 queryable/client，避免 transaction 中又回 pool 開另一條連線。

否則「看似有 BEGIN」但內部 helper 仍用全域 pool 的假 transaction 可能通過淺層 mock test。

### H-4：PUB/CX 的所有權邊界仍可能造成同一 task 由兩人切半

20.4～20.6 由 Codex 修改 `formService.js` 與 transaction，20.7 再由 Claude 接 route；20.8～20.9 Codex 重建 members service，20.10 Claude 改前端。這本身可行，但 PUB 又被描述為 Claude + Codex 混合包。

為避免 commit、review 與 tasks 勾選不清，建議把 PUB 拆成明確的序列子包：

- PUB-A（Claude）：deps、rate-limit、server infra。
- PUB-B（Codex）：validation + transaction + backend tests。
- PUB-C（Codex）：members backend + tests。
- PUB-D（Claude）：routes/front-end 接線與 integration。

每包列出所有權檔案，確保 Codex 不覆蓋 Claude 同時修改的 `server.js`／routes。

## 五、非阻斷但應在實作時驗證

### M-1：公開 members 的隱私風險已屬使用者接受的產品決策

v3 已將資料縮到 confirmed 的姓名與星等，且綁 published token。由於 published token 本質上會公開分享，持有連結者仍可取得整份 confirmed 姓名／星等名單。這不是技術漏洞，而是已定案的資料揭露邊界；上線前仍應由使用者在人眼驗收確認可接受。

### M-2：JWT 記憶體保存會在重新整理時登出

規格要求 token 只存在 React memory，因此 F5、PWA reload、頁面 crash 後都會失去 token並重新登入，不只「關頁重開」。這符合較嚴格安全姿態，但 milestone 應明確測重新整理行為，避免被誤認為 bug。

### M-3：同名重複 submissions 會影響 Table 1，但不影響 attendance boolean

v3 已定案重複提交並存。Table 1 與匯出應顯示多筆；attendance 只需判斷至少出現一次。測試應加入同名兩筆 submission，驗 attendance 的 filled 不會變成 2 或讓進度超過 total。

### M-4：現有程式碼尚未符合 v3 是預期狀態

目前 `requireAdminRole.js`、無 token `/members`、舊 `.env.example` 與 package dependency 都仍是 v1/v2 實作。tasks 已把它們列為重建／更新，不能因靜態掃描仍看到舊模式就判定 spec 未落實；真正 gate 是對應 task 完成後的測試與 diff review。

## 七、覆核結論

v3 已從「架構有重大安全漏洞」進步到「架構正確、只差規格收斂」。剩餘問題都能在 change 20 範圍內以修改 HOW 解決，不需新增 DB schema、不需另開 change，也不需要推翻既定架構。

在最小補正清單完成前：

- 可以繼續由指揮官修改計畫。
- 不建議開始 AUTH、PUB 或 CX 工作包。
- 不應執行部署、DB 操作或 dev milestone。

本次未執行 Jest、Vitest、Zeabur、LINE OAuth 或 DB 驗證；覆核結論來自文件與現有程式碼的靜態交叉檢查。
