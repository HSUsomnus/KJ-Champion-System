## Streamlit 是什麼（一段話）

Python 框架，把 Python script 直接變成互動式 data web app。特性決定了相容性邊界：它**不是靜態前端**，而是常駐 Python server（Tornado），每次使用者互動都經 WebSocket（`/_stcore/stream`）回 server 重跑 script、由 server 端驅動 UI。強項是資料表格、篩選、圖表、檔案下載等內部工具；弱項是行動端體驗、UI 深度客製、與既有 SPA 的網域整合。

## 相容性發現（分級）

### 🔴 根本衝突（無工程解，或解了等於換架構）

1. **部署架構衝突**：已定案決策「前端 Cloudflare Pages、後端 Zeabur」。Streamlit app 無法部署到 Cloudflare Pages（它是常駐 server，不是靜態產物）。UI 與 server 不可拆——採用 Streamlit 等於該頁面整組搬去 Zeabur，前端架構決策被打破。
2. **掛主網域需 WebSocket 代理**：spec 要求「後台掛在現有系統網域下」。現有 `_worker.js` 的 `proxyToBackend()` 用一般 `fetch()` 轉發，不處理 WebSocket Upgrade；要讓 `kjcs-dev.pages.dev/admin` 走到 Streamlit，需在 Worker 另寫 WebSocket 升級代理 + Streamlit 設 `server.baseUrlPath`。此組合是 Streamlit 官方 issue 與論壇的常見故障區（1006 斷線、卡「Please wait...」），屬高風險膠水工程。
3. **前台體驗衝突**：spec 核心哲學「夥伴端極簡、免登入、幾乎零打字」+ uidesign Warm Minimal 設計系統。Streamlit 元件外觀客製受限、行動端體驗平庸、初載需建立 WebSocket 連線拉整包 runtime——與「發連結給 40 位夥伴用手機按按鈕」的場景背道而馳。**前台不必再議。**

### 🟡 可解但成本高

4. **LINE Login 認證重做**：Streamlit 1.42+ 有原生 `st.login()`（OIDC via Authlib），LINE Login 是合規 OIDC provider，理論上可設為 custom provider。但：(a) session 由 Streamlit 自己的 cookie 機制管理，與 spec 甲案「4 小時 JWT、httpOnly、不落地」設計不同；(b) 登入後仍需自寫 `members.role` 角色閘門；(c) OIDC 參數客製受限（官方 issue #11703 仍開放中）。可解，但等於把已完成、測試全綠的 Section 3 重做一次。
5. **雙技術棧維護成本**：全 repo 為 Node.js + React；引入 Streamlit = 引入 Python runtime、依賴管理、測試工具（AppTest）、Zeabur Python buildpack。CCR 沙箱連 npm/node 都不可用（now.md 已知地雷），Python 同樣無法沙箱內驗證——所有驗證都推給 dev 部署與使用者本機，回饋迴圈更長。
6. **沉沒成本**：後台若改 Streamlit，Section 3 認證與 `SurveyAdmin.jsx` 骨架（含測試）作廢。

### ⚪ 誠實列出：可行的部分

7. **Zeabur 部署 Streamlit**：可行。Zeabur 支援 Python 服務，比照 `kj-survey-server` 新增服務即可。
8. **連既有 PostgreSQL**：可行。psycopg2 / `st.connection` 走 Zeabur 內網連線字串，模式與現有後端相同；唯讀查詢不違反 database skill 雙閘門（不寫入）。
9. **功能面高度對口**：Section 4–6 要做的事（全展開表格、單條件篩選、按推薦人分組進度、CSV/Excel 匯出）正是 Streamlit 的一級公民——`st.dataframe` + 篩選 widget + `st.download_button`，開發速度遠快於手刻 React 表格。**這是「想試 Streamlit」直覺正確的地方**——只是它撞上的是架構決策，不是功能適配。

## 結論與選項

| 選項 | 內容 | 代價 |
|---|---|---|
| **A（建議）** | change 20 照原 spec 續作（React 後台）；Streamlit 另開一個**實驗性 change**：獨立 Zeabur 服務、獨立網址直連（不經 `_worker.js`、不掛主網域）、**唯讀連 dev DB**，做 Table 1 + 未填名冊的內部儀表板原型，認證先用 Streamlit secrets 簡易密碼閘門（僅 dev 資料，無 prod 風險） | 多一個小型實驗 change；change 20 主線不受影響 |
| **B** | change 20 後台改用 Streamlit（獨立子網域，放棄「掛主網域」要求） | 需使用者**明確推翻**「前端架構」「部署架構」兩條已定案決策；重做 Section 3 認證（LINE OIDC via `st.login()`）；接受雙技術棧維護；`SurveyAdmin.jsx` 作廢 |
| **C** | 不引入 Streamlit，change 20 照原計畫 | 無（維持現狀） |

**建議選 A**：滿足「試試 Streamlit」的學習目的，把它放在真正適合的位置（內部資料工具），同時零風險於 change 20 主線與所有已定案決策。若原型體驗好，未來再正式評估是否讓它承接更多內部報表需求。

## 待辦清單（依使用者選擇觸發；含 Sub-agent 平行執行配置標註）

**若選 A：**

1. **change 20 復工（Sections 4–8）**——開 Sonnet `/實作` session，分支 `m_b_調查表單`。
   分工標註：**主 session 序列，不可平行**。理由（四條判準）：① Section 4–7 前端全部動 `SurveyAdmin.jsx` 同一檔案，接觸面重疊；③ 各 Section milestone 均需使用者 dev 實測互動；② ④ 依 spec 既有安排，無平行邊界可劃。
2. **Streamlit 實驗 change**——開 Opus/Fable `/規劃` session，走「新功能」流程建 `changes/29-streamlit儀表板實驗/spec.md`（編號以當時 `changes/` 最大編號 +1 為準）。範圍錨點：獨立目錄（如 `kj-survey-dashboard/`）、requirements.txt、唯讀 dev DB、secrets 密碼閘門、Zeabur dev 專案新增服務、不動 `_worker.js` 與 `frontend/`。
   分工標註：**主 session 序列**。理由：③ 規劃階段全程與使用者互動；spec 內的平行配置由該規劃 session 依判準 ④ 定案。

**若選 B：**

3. **修改計畫**——先開 `/規劃` session 改寫 change 20 spec.md（後台段全面改寫：Streamlit 服務、`st.login()` LINE OIDC 設定、獨立子網域、刪除 `SurveyAdmin.jsx` 相關 tasks），並在 spec 內明文記錄「推翻前端架構/部署架構決策，使用者已於 __ 日確認」。
   分工標註：**主 session 序列**。理由：③ 推翻已定案決策必須與使用者逐條確認；④ 新平行邊界由該次規劃定案。

**若選 C：** 僅第 1 項。
