# 後台 LINE 登入後誤入前台：診斷報告

## 交接對象

Sonnet

## 問題摘要

- 分支：`m_b_調查表單`
- dev 前端：`https://kjcs-dev.pages.dev`
- 正確後台入口：`https://kjcs-dev.pages.dev/admin`
- 發生時間：2026-07-23
- 現象：
  1. 使用無痕模式開啟 `/admin`。
  2. 畫面正確顯示「康九冠軍調查後台」及 LINE 登入按鈕。
  3. 完成 LINE 驗證後，網址變成站台根目錄 `/`。
  4. 最後顯示主系統前台首頁，而非調查表單後台。

這不是使用者誤開 `/login`；使用者已確認從 `/admin` 開始，問題仍可重現。

## 結論

高可信根因是前端 PWA Service Worker 沒有將 `/survey-api/*` 排除在 SPA navigation fallback 之外，導致 LINE OAuth callback navigation 被 Service Worker 當成前端頁面並回傳 `index.html`。

React Router 收到未知路徑 `/survey-api/admin-auth/line-callback` 後，命中 wildcard route，最後把使用者導到 `/`，因此看見主系統前台。

## 預期與實際流程

### 預期流程

```text
/admin
  → /survey-api/admin-auth/line-login
  → LINE authorize
  → /survey-api/admin-auth/line-callback?code=...&state=...
  → Cloudflare Pages Worker proxy
  → kj-survey-dev /admin-auth/line-callback
  → /admin#token=<jwt>
  → 調查表單後台
```

### 實際疑似流程

```text
/admin
  → /survey-api/admin-auth/line-login
  → LINE authorize
  → /survey-api/admin-auth/line-callback?code=...&state=...
  → PWA Service Worker navigation fallback 攔截
  → 回傳前端 index.html，callback 未進 Survey 後端
  → React Router wildcard
  → Navigate to="/"
  → 主系統前台
```

## 證據

### 1. `/admin` 路由本身正確

`frontend/src/App.jsx` 明確定義：

```jsx
{ path: '/admin', element: <SurveyAdmin /> }
```

截圖也證明 `/admin` 能正確顯示後台登入畫面。

### 2. 後台使用獨立的 Survey OAuth 入口

`frontend/src/pages/survey/SurveyAdmin.jsx`：

```js
const LINE_LOGIN_URL = '/survey-api/admin-auth/line-login'
```

因此後台沒有沿用主系統 `/api/auth/line-login`。

### 3. dev 線上登入端點產生的 `redirect_uri` 正確

2026-07-23 直接讀取線上端點：

```text
GET https://kjcs-dev.pages.dev/survey-api/admin-auth/line-login
```

回應為 `302`，`Location` 中的關鍵參數是：

```text
client_id=2008916605
redirect_uri=https%3A%2F%2Fkjcs-dev.pages.dev%2Fsurvey-api%2Fadmin-auth%2Fline-callback
```

解碼後：

```text
https://kjcs-dev.pages.dev/survey-api/admin-auth/line-callback
```

所以登入入口、LINE `redirect_uri` 與 dev origin 都正確，不應先去修改 LINE Channel、Survey 後端 `FRONTEND_URL` 或 callback route。

### 4. Survey 後端成功與失敗都固定返回 `/admin`

`kj-survey-server/routes/adminAuth.js`：

```js
const redirectWithError = (res, code) =>
  res.redirect(`${env.FRONTEND_URL}/admin?authError=${code}`)

res.redirect(`${env.FRONTEND_URL}/admin#token=${token}`)
```

後端程式沒有成功後導向 `/` 或 `/login` 的路徑。

### 5. Cloudflare Pages Worker 已有 `/survey-api/*` proxy

`frontend/public/_worker.js` 已處理：

```js
if (url.pathname.startsWith('/survey-api/')) {
  // proxy 到 kj-survey-server
}
```

只要 callback request 到達 Pages Worker，便應進入 Survey 後端。

### 6. PWA navigation fallback 只排除了主系統 `/api/*`

`frontend/vite.config.js` 目前為：

```js
workbox: {
  navigateFallbackDenylist: [/^\/api\//],
  runtimeCaching: [
    {
      urlPattern: /^https?:\/\/.*\/api\/.*/,
      handler: 'NetworkFirst',
      // ...
    },
  ],
}
```

缺少 `/survey-api/*`。

OAuth callback 是瀏覽器的頂層 navigation。Service Worker 若把它視為 SPA navigation fallback，就會回傳 `index.html`，而不是讓 request 到 `_worker.js`。

### 7. React Router 的 wildcard 行為與實際結果吻合

`frontend/src/App.jsx`：

```jsx
{ path: '*', element: <Navigate to="/" replace /> }
```

因此 callback 被當成 SPA 頁面時，最終網址與畫面都會變成 `/`；這與使用者截圖完全吻合。

## 為什麼無痕模式仍會發生

無痕模式不會禁止網站註冊 Service Worker。使用者先開啟 `/admin` 時，該無痕 session 仍可能安裝並啟用本站 Service Worker；從 LINE 返回 callback 時，navigation 就可能受到它控制。

所以「無痕模式仍重現」不能排除 Service Worker，反而與本問題流程相容。

## Sonnet 修復前的必要確認

先做一次對照實驗，以 DevTools 證實 callback 是否被 Service Worker 攔截：

1. 開啟 Chrome DevTools。
2. 到 Application → Service Workers。
3. 勾選 `Bypass for network`，或只在本次測試 unregister。
4. 從 `/admin` 重新完成一次 LINE 登入。
5. 若停用／繞過 Service Worker 後能回到 `/admin`，即可確認根因。
6. 在 Network 面板確認 callback：
   - Request URL 是 `/survey-api/admin-auth/line-callback?...`
   - 修復前可能由 Service Worker 回傳前端 HTML。
   - 正常時應經 Pages Worker／Survey 後端返回 `302 /admin#token=...`。

注意：unregister 只用於診斷對照，不是正式修復。

## 建議的最小修復範圍

主要修改檔案：

```text
frontend/vite.config.js
```

至少將 `/survey-api/*` 加入 `navigateFallbackDenylist`：

```js
navigateFallbackDenylist: [
  /^\/api\//,
  /^\/survey-api\//,
],
```

並檢查 `runtimeCaching` 規則是否也需要明確處理 `/survey-api/*`。原則是所有 `/survey-api/*` 請求都必須走網路與 Cloudflare Pages Worker proxy，不可回傳 SPA `index.html`，也不應使用會掩蓋 OAuth redirect 的快取回應。
