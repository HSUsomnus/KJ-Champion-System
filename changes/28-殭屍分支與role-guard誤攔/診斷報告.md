- 觸發物：使用者截圖（DevVps 執行 `git push origin main && bash scripts/sync-branches.sh`，`m_b_role-guard補強` 回報 `conflict-stopped`、behind main 2）

## 結論（TL;DR）

截圖中的 `conflict-stopped` **不是 script 壞掉，也不是內容遺失**——`sync-branches.sh` 行為完全符合 change 22 設計（衝突即停、交人工）。真正的問題有兩個：

1. `m_b_role-guard補強` 是**殭屍分支**：change 27 內容已 100% 在 main 上（以 squash 直推 commit `b5f6231` 上線），分支只剩 21 顆同內容、不同 SHA 的歷史 commit。該刪，不該 merge。
2. 診斷過程中實測到 **role-guard.js doctor 段誤攔唯讀指令 `git merge-base`**——與 change 27 剛修完的 `\bmain\b` push-main 誤攔是同型 bug，規則層需要補一刀。

---

## 發現清單

### 🟠 懸掛 1：`m_b_role-guard補強` conflict-stopped（截圖主角）

**現況**：sync 時 merge main 衝突而停下，behind main 2（診斷當下已變 3，因 main 又進了 `dc96a37` 收尾 commit）。

**根因**：change 27 走「規則類直推」——實作內容被打包成單一 squash commit `b5f6231` 直推 main，但功能分支上原本的 21 顆細粒度 commit（Phase 1–5 + change 26 繼承歷史）仍留在遠端。結果是**同一份內容、兩條不同歷史**：

- 自 merge base（`002cb01`）起，main 側與分支側動的是**完全相同的 8 個檔案**（role-guard.js、session-role-notice.js、settings.json、實作.md、vps新對話.md、now.md、change 27 spec/tasks）。
- 內容相同的檔案 git 可自動合併；**唯一實質衝突只有 `changes/27-role-guard補強/tasks.md` 的 5.4 / 5.5 兩行**（main 已勾選並補註完成說明，分支停在未勾），且 main 版本才是事實。
- main 上的 tasks.md 5.5 其實已白紙黑字寫了「`m_b_role-guard補強` conflict-stopped 屬預期，該分支已完成階段性任務」——這是**已知的待清理項**，不是意外。

**建議做法**：直接刪除遠端分支 `m_b_role-guard補強`，不做人工 merge（merge 只會為一條死分支製造 merge commit）。刪除前不需搬任何東西——分支 vs main 的全部差異就是 tasks.md 那兩行未勾選，main 為準。

**風險/取捨**：無內容損失風險（已逐檔 diff 驗證）。若想保留細粒度歷史可先打 archive tag，但 change 27 屬規則類小改、spec 與 context 檔已承載脈絡，個人判斷不需要。

### 🟠 懸掛 2：`m_b_打包繼續指令` 同為殭屍分支，正在被 sync「養著」

**現況**：change 26 已上線 main，但分支還在。這次 sync 剛好無衝突（`6104930..0b55fe7`），於是被自動 merge + push——**每跑一次 sync 就多一顆無意義的 merge commit**。目前 vs main 的內容差異也只剩 tasks.md 一行勾選。

**根因**：與懸掛 1 相同——「規則類直推」上線後沒有刪功能分支。workflow skill 的「功能上線」關鍵字本來就定義為「merge 到 main，**刪分支**」，但規則類直推路徑沒有對應的刪分支步驟，分支就懸掛下來了。

**建議做法**：一併刪除遠端分支 `m_b_打包繼續指令`。

### 🟡 過時 3：兩條 CCR 舊 session 分支（`claude/*`）

**現況**：
- `claude/change-20-strategy-hf8eds`：落後 main 14 commit，是 change 20 規劃期的 CCR session 分支；change 20 spec 已遷入 `m_b_調查表單` 的 `changes/20-…/spec.md`。
- `claude/packing-continue-skills-audit-qr0c9z`：落後 main 14 commit，是 change 26 規劃期分支；change 26 已上線。

**建議做法**：確認無未搬內容後刪除（`claude/change-20-strategy-hf8eds` 建議由接手 change 20 的 session 順手比對一眼再刪，因 change 20 尚未復工）。

### 🔴 壞掉 4：role-guard.js doctor 段誤攔唯讀 `git merge-base`

**現況**：本診斷 session 實測，執行含 `git merge-base` 的唯讀查詢被 role-guard deny。

**根因**：`.claude/hooks/role-guard.js:87` 的 `/^git merge\b/`——`\b` 在 `merge-base` 的 hyphen 處成立，因此 `git merge-base`、`git merge-tree`（皆唯讀）都會中招。這與 change 27 Phase 1 剛修掉的 engineer 段 `\bmain\b` 誤攔（`fix-main-layout` 被攔）是**同一型 regex bug**，只是這次長在 doctor 段。/診斷 指令規格明文「唯讀 git 操作不受限」，此為違反規格的誤攔。

**建議做法**：改為 `/^git merge(\s|$)/`（hyphen 不再匹配），或比照 change 27 的 token 比對寫法統一整段風格。順手檢查同段其餘 regex（`git branch\b.*\s-[dD]\b` 等）有無同型問題。

**影響面**：只影響 doctor 角色 session 的診斷效率（被迫繞路），不影響產品與其他角色。嚴重度低、修複成本極低。

### ⚪ 債務 5：「規則類直推」與功能分支並行的制度缺口

**根因層面的通則**：只要「在功能分支上開發 → 內容以 squash 直推 main 上線」這條路徑存在，就**必然**產生同內容雙歷史的殭屍分支，而 `sync-branches.sh` 對所有 `m_b_*` 一視同仁，殭屍分支不是每次 conflict-stopped 干擾判讀（懸掛 1），就是被靜默灌 merge commit（懸掛 2）。

**建議做法（擇一或並行，交規劃討論）**：
- A（建議，改流程）：deploy-release skill「規則類直推」段增訂收尾步驟——直推上線驗證後，同一批自包含指令內附上 `git push origin --delete <分支>`，與 workflow「功能上線＝merge + 刪分支」對齊。
- B（改工具）：`sync-branches.sh` 增加退役清單（如讀 `.claude/sync-skip.txt`）跳過已交付分支。A 已足夠的話 B 屬過度設計。

### （附註）非問題項

- `sync-branches.sh` 本身：衝突即停、abort、回報、behind 驗證，全部按 change 22 設計運作，**無 bug**。
- `m_b_調查表單`（change 20，待復工）：本次 sync 成功、內容為真實未上線功能（kj-survey-server 約 8k 行），**必須保留**。
