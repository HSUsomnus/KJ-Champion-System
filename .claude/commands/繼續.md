# 繼續

## 定位

新對話的接手指令。先讀狀態、再回報、才接受使用者的下一步指令——與 CLAUDE.md 對話啟動規則相容、且更完整（多了交接檔讀取）。

## 步驟 1：檢查工作區是否已有交接檔

```bash
ls .claude/handoff.md
```

若存在（使用者已照 `/打包` 給的起手式 `git fetch` + `git checkout` 切到工作分支）→ 直接讀 `.claude/handoff.md`，跳到步驟 3。

## 步驟 2：找不到工作區交接檔 → fallback 搜尋

```bash
git fetch --all --quiet
git log --all -1 --format='%H %ct %D' -- .claude/handoff.md
```

- **找到 commit** → checkout 該 commit 所在分支，讀該版本的 `.claude/handoff.md`。
  - 若該 commit 其實是「刪除 handoff」那次 commit（即 handoff 在該版本已不存在，只是最後一次改動它的 commit）→ 視同找不到，走下一條。
- **找不到，或找到的 commit 已是刪除動作** → fallback：讀 `.claude/now.md` 的「當前 Change」段，向使用者回報目前 now.md 記錄的狀態，**明確詢問使用者要接手什麼**（哪個 change、哪個分支）。不得自行猜測或逕自挑一個 change 開始動手。

## 步驟 3：讀取上下文

讀以下檔案（存在的都讀）：

- `.claude/now.md`
- 該 change 的 `changes/NN-名稱/spec.md`
- 該 change 的 `changes/NN-名稱/tasks.md`

## 步驟 4：回報接手摘要

向使用者回報，內容至少包含：

- 上次做到哪（依 handoff 的「已完成 / 進行中 / 下一步」，或 now.md 記錄）
- 目前工作區與分支狀態（`git status --short`、`git branch --show-current`）
- 下一步建議做什麼

## 步驟 5：打卡角色

依 handoff 記錄的角色提醒使用者打卡：

- 角色為 planner → 提醒輸入 `/規劃`
- 角色為 engineer → 提醒輸入 `/實作`
- handoff 不存在或未記錄角色（fallback 走 now.md 那條）→ 詢問使用者這個 session 要用哪個角色

不自行呼叫 `/規劃` 或 `/實作`，由使用者確認後執行。

## 步驟 6：交接檔生命週期＝單次

接手後的**第一個 commit**，順手一併刪除 `.claude/handoff.md`：

```bash
git rm .claude/handoff.md
```

與該次功能改動放進同一個 commit（或若該次沒有功能改動要 commit，另開一個小 commit 專門刪除）。

⛔ 不可讓 `.claude/handoff.md` 殘留超過一次接手——陳舊的交接檔會誤導下一次 `/繼續`，比沒有交接檔更危險（比照 now.md「地雷清單只增不減即膨脹失效」的教訓）。
