# v2.0.6 — Login.jsx useEffect 與 handleConfirm 的 navigate race condition

**Tag**: v2.0.6
**Date**: 2026-04-25
**Type**: hotfix（v2.0.5 後續修補）
**Branch flow**: main → hotfix/login-no-profile-useeffect-race → main

## 背景

v2.0.5 修復「`'no-profile'` 分支沒呼叫 `login(userData)` 導致 ProtectedRoute 踢回 /login」後，使用者重測時撞到第二層 bug — 點「建立資料」**沒進 `/profile/edit`，反而跑去首頁**。

Network log 顯示 `events?startDate=...` 200 — 是首頁的請求，證明使用者最終停在 `/`。

## 為什麼 v2.0.5 沒修徹底

v2.0.5 在 `handleConfirm` 補了 `login(userData)`，意圖是讓 user state 不為 null 通過 ProtectedRoute。但**沒注意到 Login.jsx 第一個 useEffect 的存在**：

```js
useEffect(() => {
  if (user) navigate('/', { replace: true })
}, [user, navigate])
```

這段是「已登入直接訪問 /login → 跳首頁」邏輯，原本只在 user 從 localStorage 還原時觸發。

但 v2.0.5 的修法在 `handleConfirm` 裡呼叫 `login(userData)` → setUser → user 變更 → 此 useEffect 重新觸發 → `navigate('/')`，蓋掉同步呼叫的 `navigate('/profile/edit')`。

實際時序：

```
t0: handleConfirm 呼叫
t1: login(userData) → React schedule setUser
t2: navigate('/profile/edit') → 路由切到 /profile/edit
t3: React flush state update → user 變更
t4: 元件 re-render → useEffect 因 user 變更重新跑
t5: useEffect: if (user) navigate('/') → 路由切到 /
t6: 最終停在 /
```

## 根因

useEffect 的觸發條件太寬。它只看 `user`，不看 `authState`，所以 OAuth callback 流程改了 user 之後也會被觸發。

## 修法

useEffect 加 `authState === 'idle'` 條件，把觸發限縮到「初始進 /login 時 user 已存在」的情境：

```js
useEffect(() => {
  if (user && authState === 'idle') navigate('/', { replace: true })
}, [user, navigate, authState])
```

### 為什麼 `authState === 'idle'` 是對的判斷

`authState` 的生命週期：

- 進入 /login（有 localStorage userId）→ `idle`
- AuthContext 拿 user → user 變更 → useEffect 觸發 → 跳首頁（保留 v2.0.6 之前的行為）
- OAuth callback 回來（URL 帶 userId）→ `idle → loading → has-profile / no-profile`
- handleConfirm 呼叫 `login(userData)` → user 變更，但此時 authState 不是 `idle`（已是 has-profile / no-profile）
- useEffect 不觸發，handleConfirm 的 navigate 生效

## 教訓

v2.0.5 修補時只看 handleConfirm 的 else 分支，沒通讀整個 Login.jsx 元件的 lifecycle。React 元件中**任何** state 變更都可能觸發 dependent useEffect，動 user state 必須一併檢查所有依賴 user 的 useEffect。

下次修 React 元件 bug 流程：

1. grep 元件內所有 `useEffect`
2. 列出 deps 含被你修改 state 的所有 effects
3. 評估每個 effect 在新邏輯下會不會誤觸發
4. 必要時加守門條件（如本次的 `authState === 'idle'`）或拆分 effect
