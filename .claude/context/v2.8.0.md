# v2.8.0 — change 17「SidebarNav UI 重構」

## 背景

v2.5.0 上線的 SidebarNav 雖然已整合 Header + FabNav，但存在以下缺口：
1. SidebarNav 頂部只有 logo 圖，沒有品牌文字，視覺空洞
2. 管理者後台（/management）沒有導覽入口，負責人找不到入口
3. Management.jsx 的 tab 樣式與 Profile.jsx / MemberDetail.jsx 的 pill 風格不統一
4. 只有開發者可以在 /management 修改成員角色，負責人無法操作
5. UIDESIGN.md 缺乏 Pill Tab 與 SidebarNav 的明確規範

同時 v2.7.0 的 admin.js 有一個安全問題：sync-backup-to-dev 是定時 + API 觸發寫入 dev DB，與使用者「dev DB 只能手動寫入」的安全原則衝突。本版一併修正。

## 關鍵設計決策

### isManager = role !== '一般人'（而非 role === '管理者'）

系統目前角色分四種：一般人 / 管理者 / 負責人 / 開發者。
管理者後台入口的可見性應該是「不是一般人都能看到」，而非只有「管理者」role 字串才能看到。
用 `role !== '一般人'` 比 `['管理者','負責人','開發者'].includes(role)` 更簡潔，且新增角色時不需要修改邏輯。

### 移除 sync-backup-to-dev 而非禁用

API 若保留，將來可能被誤呼叫（定時排程 + 手動觸發都存在）。
直接從 admin.js 移除比加 guard 更安全 — 「不存在」比「有但不能用」更可靠。
改以 export-backup-csv（prod 後端走內網取資料）+ import-csv-to-dev.js（本機 UPSERT）的兩步驟手動流程取代。

### backup DB 不需開公網的架構

export-backup-csv API 在 prod 後端（kj-champion 專案）執行，可以走 `postgresql-backup.zeabur.internal` 內網取資料，再以 CSV 回傳給呼叫方。呼叫方只需要有 ADMIN_SECRET，不需要 backup DB 公網開放。dev DB 的寫入由使用者下載 CSV 後在本機跑 import-csv-to-dev.js，全程 backup DB 不暴露公網。

## 學習日誌

### 問題 1：Zeabur dev 後端連續兩次部署失敗（建置日誌空白）

**遇到的問題**：merge m_b_SidebarNav_UI 到 dev 後，Zeabur kj-champion-dev 服務連續兩次「部署失敗」，建置日誌完全空白（找不到日誌）。

**試過的方法**：點「重新部署」→ 再次失敗；查看「運作紀錄」→ 無輸出。

**最終解法**：判斷為 Zeabur 基礎建設問題（build container 沒有啟動，非程式碼錯誤）。直接砍掉遠端 dev 分支，從 main 重新切出乾淨的 dev，merge 功能分支，push 新 dev → Zeabur 成功部署。

**根本原因**：推測是 Zeabur build queue 異常或舊部署殘留狀態導致 container 啟動失敗。重建分支強制觸發全新 webhook 事件，繞過了問題。dev 重建無損失（dev 是拋棄式測試環境）。

### 問題 2：複製 backup DB 資料到 dev DB 的網路限制

**遇到的問題**：最初計畫直接從後端 API 或本機腳本連 backup DB 取資料再寫入 dev DB。但 backup DB 在 kj-champion（prod）專案內網，dev 後端在 kj-champion-dev 專案，跨專案內網不通；backup DB 公網也是關閉的（安全設計）。

**試過的方法**：dev 後端 API 連 backup DB → ENOTFOUND；Zeabur 網頁 SQL 介面 → 需要公網連接埠；本機 Node.js 腳本 → 需要公網；psql pipe 方式（在 backup DB container 終端機執行 psql | psql DEV_URL）→ 成功但只適合一次性複製，不適合重複使用的流程。

**最終解法**：prod 後端新增 export-backup-csv API（走內網取資料 → 回傳 CSV），搭配本機 import-csv-to-dev.js（讀 CSV → UPSERT dev DB）。兩步驟完全不需要 backup DB 開公網。

**根本原因**：Zeabur 跨專案內網隔離是刻意設計的安全邊界，不應繞過。正確解法是讓有權限的服務（prod 後端）做資料中繼，而非直接打開 DB 公網。
