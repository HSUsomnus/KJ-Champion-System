## 背景

前台行事曆資料來源是本地 DB（PostgreSQL），前端打 `/api/calendar` 讀 DB 不直接打 Google Calendar。這造成一個問題：管理員在 Google Calendar 新增/修改/刪除行程後，前端看不到更新，除非後端主動同步。

舊做法：使用者手動操作（前端 CRUD）才會同步一次。沒有 Google Calendar → DB 的自動同步機制，Calendar 和 DB 容易脫鉤。

本版目標：加入 node-cron 每分鐘自動同步，讓 Google Calendar 的變更在 1 分鐘內反映到系統前端，不需人工介入。

## 關鍵設計決策

### 1. `/api/debug` 永遠掛載，不受 NODE_ENV 限制

原先用 `if (process.env.NODE_ENV !== 'production')` 保護。Zeabur DEV 環境 `NODE_ENV=production`，導致 debug 端點在 DEV 就 404，完全無法用來驗證部署。移除 guard，改為「只要在 Zeabur 就能打」——因為這個端點只回傳狀態，不修改資料，風險可接受。

### 2. Token cache 在 module level，不在 request level

`_tokenCache` 掛在 module scope，跨所有請求共用。原因：Service Account token 有效 1 小時，每次 Calendar API 都換一個新 token 是浪費，也可能觸發 Google 的 rate limit。Module-level cache 確保整個 process 生命週期內最多每 55 分鐘換一次 token。

### 3. `token_uri` 讀自 JSON，不硬編碼

Service Account JSON 裡本來就有 `token_uri: "https://oauth2.googleapis.com/token"`。改成讀 `credentials.token_uri || TOKEN_URL`，讓端點跟著金鑰走。未來 Google 如果再改端點，換 JSON 就好，不用改程式碼。

## 學習日誌

### 問題一：Token 換取失敗 — `Premature close` at `www.googleapis.com/oauth2/v4/token`

**遇到的問題**

Zeabur DEV 日誌：`TypeError: fetch failed — reason: Premature close` 指向 token 端點。

**試過的方法**

1. **JWT constructor 加 `tokenUrl: TOKEN_URL`** — 無效。`google-auth-library@9.x` `JWT.refreshTokenNoCache()` 內部硬編碼舊 URL，constructor option 根本沒被讀到。

2. **覆寫 JWT instance 的 `getAccessToken`、`authorize`、`getRequestHeaders`** — 無效。`googleapis-common` 呼叫 `auth.request()` → `OAuth2Client.refreshTokenNoCache()`，走的是另一條路，instance override 完全繞不進去。

3. **換用 `google.auth.OAuth2` + `setCredentials()`** — token 換取錯誤消失，但 Calendar API 呼叫冒出新的 Premature close（見問題二）。

**最終解法**

完全不走 `google-auth-library`。自己用 `crypto.createSign('RSA-SHA256')` 簽 JWT assertion（`aud` 設正確的 `oauth2.googleapis.com/token`），再用 `https.request` POST 換 token。

**根本原因**：`JWT.refreshTokenNoCache()` 的 `aud` 欄位硬編碼舊廢棄端點 `www.googleapis.com/oauth2/v4/token`。新端點驗 `aud` 不符就直接關 TCP → Premature close。覆寫任何 instance method 都無法改 `aud`，因為 assertion 在內部生成。

---

### 問題二：Calendar API 呼叫失敗 — `Premature close` at `www.googleapis.com/calendar/v3/...`

**遇到的問題**

問題一解決後，Calendar API 呼叫仍然失敗（不同的 URL）。同時 `/api/debug/health`（一直用 raw `https.request`）從頭到尾正常。

**試過的方法**

1. **繼續用 `googleapis` 的 `calendar.events.list()` 等** — 全部走 gaxios → undici → Premature close，方向錯誤。

**最終解法**

`calendarService.js` 全面改寫，所有 Calendar API 呼叫改用封裝好的 `calendarApiRequest()`（raw `https.request`），完全不走 `googleapis` / `gaxios`。

**根本原因**：`gaxios@6+` 在 Node.js 18 偵測到 `globalThis.fetch`，改用 `undici`（native fetch）。`undici` 對 TCP 半關閉敏感：Zeabur 容器到 Google API 之間的 NAT 有時提前關 socket，`undici` 就拋 Premature close；Node.js 內建 `https.request` 比較寬鬆。`debug.js` 從一開始就用 `https.request` 所以一直正常，這是找到正確方向的關鍵依據。
