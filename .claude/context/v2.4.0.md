## 學習日誌

### 問題一：Token 換取失敗 — `Premature close` at `www.googleapis.com/oauth2/v4/token`

**遇到的問題**

Zeabur DEV 日誌顯示：

```
TypeError: fetch failed — reason: Premature close
  at https://www.googleapis.com/oauth2/v4/token
```

**試過的方法**

1. **在 JWT constructor 加 `tokenUrl: TOKEN_URL`**
   — 無效。`google-auth-library@9.x` `JWT.refreshTokenNoCache()` 內部硬編碼舊 URL，constructor option 沒作用。

2. **覆寫 JWT instance 的 `getAccessToken`、`authorize`、`getRequestHeaders`**
   — 無效。`googleapis-common` 呼叫 `auth.request()`，走的是 `OAuth2Client.refreshTokenNoCache()`，instance method override 完全繞不進去。

3. **換用 `google.auth.OAuth2` + `setCredentials()`**（避開 JWT）
   — token 換取錯誤消失，但 Calendar API 呼叫出現新的 `Premature close`（見問題二）。

**最終解法**

完全不走 `google-auth-library`。自己用 `crypto.createSign('RSA-SHA256')` 簽 JWT assertion（`aud` 設正確的 `oauth2.googleapis.com/token`），再用 `https.request` POST 換 token：

```js
// server/config/googleAuth.js
const exchangeJWTForToken = (credentials) =>
  new Promise((resolve, reject) => {
    const tokenUrl = credentials._tokenUrl || TOKEN_URL;
    // 自組 JWT header.payload.sig，aud = tokenUrl（正確端點）
    // https.request POST 到 oauth2.googleapis.com/token
  });
```

**根本原因**：`JWT.refreshTokenNoCache()` 的 `aud` 欄位硬編碼舊廢棄端點 `www.googleapis.com/oauth2/v4/token`。新端點驗 `aud` 不符就直接關 TCP → Premature close。覆寫任何 instance method 都無法改 `aud`，因為 assertion 在內部生成。

### 問題二：Calendar API 呼叫失敗 — `Premature close` at `www.googleapis.com/calendar/v3/...`

**遇到的問題**

Token 換取解決後，Calendar API 呼叫仍然失敗：

```
TypeError: fetch failed — reason: Premature close
  at https://www.googleapis.com/calendar/v3/calendars/.../events?...
```

`/api/debug/health` 端點（一直用 raw `https.request`）沒有這個問題。

**試過的方法**

1. **繼續用 `googleapis` 的 `calendar.events.list()` 等**
   — 全部走 gaxios → undici → Premature close，無論如何無解。

**最終解法**

`calendarService.js` 全面改寫，所有 Calendar API 呼叫改用封裝好的 `calendarApiRequest()`（raw `https.request`），完全不走 `googleapis` / `gaxios`：

```js
// 舊（壞）
const calendar = await getCalendarClient();
await calendar.events.list(...)  // → gaxios → undici → Premature close

// 新（好）
const token = await getToken();
await calendarApiRequest({ method: 'GET', path: `/calendar/v3/...`, token })  // → https.request → 正常
```

**根本原因**：`gaxios@6+` 在 Node.js 18 環境偵測到 `globalThis.fetch` 存在，改用 `undici`（Node.js 18 native fetch 底層）。`undici` 對 TCP 半關閉（half-close）比 `node-fetch` 敏感：Zeabur 容器到 Google API 之間的 NAT 有時會在 HTTP 回應送完前先關 TCP socket，`undici` 偵測到就拋 Premature close；Node.js 內建的 `https.request` 比較寬鬆，正常完成。

**驗證方式**：`debug.js` 從一開始就用 `https.request`，health check 一直 `ok: true`。這是改對方向的決定性依據。
