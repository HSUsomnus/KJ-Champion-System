## 背景

change 23 建立了三層角色系統（規劃 → 實作 → 收尾員），change 24 進行第一次實地驗證並暴露了六個交接缺口。本版依 change 24 的驗證發現，系統化地修補這些缺口，使三層流程的交接訊息、機制、與收尾規則更加完善。此外，change 12「統一彈出訊息系統」已由使用者確認廢除，本版同步更新了 now.md 的記錄。

## 改動檔案

- **`.claude/commands/規劃.md`**：停止點交接訊息新增兩個必要資訊（spec 分支位置、起手式三行）、停止點前加「確認閘」機制——未獲使用者明確確認前禁止後續 commit / push / 開 PR，避免規劃清單變動後規劃快照遺失
- **`.claude/commands/實作.md`**：加入「spec 取得」段落（找不到 spec 先問使用者在哪個分支，不得自行猜測）；收尾筆記結構從四段改為五段（插入「主 session 連結」作為獨立段，方便收尾員取值）；每個 Phase commit 後立即 push 遠端（CCR 沙箱生命週期短，push 才是真正的異地 checkpoint）
- **`.claude/agents/收尾員.md`**：交接素材清單與 commit trailer 範本改為自收尾筆記動態取值（主 session 連結、版本號、改動摘要、學習日誌、驗證結果），缺連結時明寫 fallback 字串而非自行編造 URL
- **`.claude/skills/deploy-release/SKILL.md`**：CCR 403 fallback 補做指令改為自包含範本（含明確 merge commit SHA，假設使用者在全新 clone 執行而無本地狀態）；新增「SSH 簽章與本地驗證（CCR 沙箱）」小節說明 git log 顯示 N 是缺 allowedSignersFile 無法本地驗證、不是沒簽章，禁止誤判後重寫歷史
- **`.claude/now.md`**：CCR 403 地雷條目補「補做指令必須自包含」；新增「簽章顯示 N 屬正常」地雷條目；change 12「統一彈出訊息系統」記錄從「0/33 尚未開始」改為正確的封存廢除狀態，成品在 tag `archive/change-12` 可撿回

## 關鍵設計決策

1. **確認閘的位置與逆序執行**：確認閘設置在規劃停止點，但執行 commit / push 檢查是由 git-guard.js hook 在 git 指令時逆序檢查「主 session 是否給過確認訊息」。此設計允許實作員不用文字確認、直接依 hook 回報決定是否続行，避免增加實作流程的中斷點。
2. **Phase 即時 push 的強制性**：CCR 沙箱是無狀態短命環境，本地分支無法作為 checkpoint。規則改為「每 Phase commit 後立即 push 遠端」而不是「等功能完成再統一 push」，在沙箱環境中保證每個 Phase 的工作成果不會因為沙箱重啟而遺失。
3. **收尾筆記的「主 session 連結」獨立段**：為了讓收尾員能準確取值而不依賴文字格式解析，特別開闢獨立段落放主 session 連結，同時要求缺連結時輸出 fallback 字串而非允許自由編造 URL（避免回溯錯誤）。
4. **補做指令的自包含原則**：tag 推送的 403 fallback 指令給使用者時，不能假設使用者有本地分支或本地 tag，必須同時提供明確的 SHA 讓使用者先建立 tag 再推送，這樣即使沙箱完全失效該指令在全新 clone 上也能執行。
5. **禁止為簽章顯示 N 而重寫歷史**：在 CCR 沙箱中 commit 有簽但本地驗證失敗是環境限制而非真實問題，GitHub 上仍會顯示 Verified badge。規則明確禁止為此重寫歷史，以 GitHub badge 為唯一檢查來源。

## 學習日誌

### 問題：Phase 3 gate「自包含」grep 計數與 spec 預期值不符

**遇到的問題**：spec 8.3 gate 預期 `grep -c "自包含" .claude/skills/deploy-release/SKILL.md` ≥2，實測完全依 spec 3.1 原文加入內容後，結果只有 1（`grep -c` 是計算「符合的行數」而非文字出現總次數，該段落內「自包含」一詞在 spec 提供的原文裡本來就只出現一次）

**試過的方法**：
- 重新核對 spec 5-3.1 全文，確認並未遺漏任何段落
- 核對 5-3.2（SSH 簽章小節）也未包含「自包含」一詞，兩段合計仍是 1

**最終解法**：判定為顧問撰寫 spec 時的 gate 預期值估算誤差（非內容缺漏），比照 workflow skill 已記載的 change 22 先例（「gate 7.3 的 grep 計數字面上不可能達成」）處理方式——不阻塞、不擴大範圍硬湊字數，忠實依 spec 原文落實內容後繼續，於 tasks.md 與本筆記如實記錄差異

**根本原因**：spec 撰寫階段雖已實測「門檻 ≥2」，但「預期值」是顧問人工估算文字改動後的計數，未在下筆前先草擬好完整替換文字再實際跑一次 grep 驗證預期值。導致估算與實際新增文字的字面出現次數有落差。（根本教訓：gate 須在 spec 草稿時先實測出真實計數，不應人工估算。此流程改進列入 workflow skill 記載。）
