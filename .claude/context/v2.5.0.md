# v2.5.0 — change 14「側邊欄導覽」

## 背景

使用者希望改善行動裝置上的導覽體驗。原本系統有兩個導覽元素：固定在頂部的 Header bar（Logo + 重整 + 用戶頭像）和固定在左下角的 FabNav（漢堡按鈕，展開時往上彈出導覽項目）。使用者覺得上下兩個固定元素佔用空間且體驗分散，希望整合成一個單一的左側抽屜式側邊欄。

## 關鍵設計決策

**Layout 包裹器而非每頁各自引入**：SidebarNav 只在 Layout.jsx 宣告一次，App.jsx 以 `{ element: <Layout />, children: [...] }` 包住所有受保護路由。好處是未來新增頁面只需在 App.jsx 的 children 陣列加一行路由，不需要在頁面檔案裡寫任何導覽相關程式碼。

**createPortal → document.body**：SidebarNav 的漢堡 FAB、遮罩、抽屜全部用 createPortal 掛到 document.body。原因是避免頁面元件的 stacking context（z-index 層疊）影響固定定位元素的顯示順序。舊的 FabNav 也是同樣做法。

**保留 FabAction 不動**：右側浮動操作按鈕（FabAction）是頁面專屬的操作入口（如儲存、編輯），與全域導覽性質不同，每個頁面自己宣告。

## 學習日誌

### 問題 1：App.jsx 巢狀結構寫錯

**遇到的問題**：第一次 Edit 嘗試加入 Layout 層時，router 陣列的括號配對出錯，導致整個路由結構 malformed，頁面空白。

**試過的方法**：分段 Edit 加括號 → 仍出錯（局部 Edit 不易掌握整體括號平衡）。

**最終解法**：整個 `createBrowserRouter([...])` 區塊重寫，一次輸出完整結構。

**根本原因**：JSX 的 children 陣列巢狀多層時，括號對數難以在局部 diff 中追蹤。

### 問題 2：SidebarNav 重複 borderTop key

**遇到的問題**：profile 按鈕的 style object 有 `{ borderTop: '...', border: 'none', borderTop: '...' }` 重複 key，React 會忽略第一個，視覺上 border 沒套用正確。

**最終解法**：改為 `{ border: 'none', borderTop: '1px solid #E2DED8' }` 去掉重複。

**根本原因**：手寫 inline style object 時複製粼接錯誤。

### 問題 3：merge dev 衝突 — FeedbackProvider / toast hooks

**遇到的問題**：merge m_b_側邊欄導覽 → dev 時，dev 另外有 change 12 的 FeedbackProvider 加入（App.jsx）及 toast/confirm hooks（AddEvent / EventDetail / Management）。我們的分支移除了 Header/FabNav，dev 加了 feedback 系統，兩邊在不同地方改同一批檔案。

**最終解法**：手動 resolve 4 個衝突檔案，保留 FeedbackProvider + toast/confirm（dev 獨有功能），保留 Layout + 移除 Header/FabNav（我們的變更）；main merge 時 fast-forward 無衝突（main 沒有 FeedbackProvider）。
