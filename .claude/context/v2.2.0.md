# v2.2.0 — 每日行程推播 LINE Bot（後端）

**Tag**: v2.2.0
**Date**: 2026-04-25
**Type**: feature（新增功能 — OpenSpec change 09 後端部分）
**Branch flow**: m_b_每日行程推播_backend → main

## 背景

社團行程過去只能在前端行事曆查看，成員容易錯過。需求是：每天固定時間 LINE Bot 推送「明日行程」Flex 字卡到目標成員，可由開發者調整時間 / 啟停 / 對象。

開發在 `m_b_每日行程推播_backend` 分支進行（手機 Claude Code Web 與 PC 雙裝置接力），合進 dev 後連續多日 23:30 推播驗證視覺與時區行為都正確（注意：dev 後端在 v2.1.0 之前連的是 prod DB，所以實機名單是 prod 真實成員 — 等於 prod-data × code 已預先驗證過）。

## 架構

```
node-cron task (Asia/Taipei timezone, 預設 21:00)
  ↓
agendaService.sendDailyAgenda()
  ├── eventDbService.getEventsByDate(tomorrow)            ← 讀隔日行程
  ├── memberDbService.getMembers()                         ← 讀成員清單
  ├── filterMembersByTarget()                              ← 依對象過濾角色
  ├── lineService.generateDailyAgendaFlexMessage(events)  ← 產生 Flex JSON
  └── lineService.pushFlexMessage(lineUserId, flex)       ← 逐人推送
```

設定來源是 `system_settings` 表（key/value/updated_at），三筆預設：

| key | 預設值 | 說明 |
|---|---|---|
| `daily_agenda_time` | `21:00` | 推播時間 HH:MM（Asia/Taipei） |
| `daily_agenda_enabled` | `true` | 啟停 |
| `daily_agenda_target` | `developer` | 對象（all / manager_above / developer） |

## 時區處理（5.5.1 修補的 bug）

最早版本用 `new Date(date.toLocaleString('zh-TW', { timeZone: 'Asia/Taipei' }))` 計算明日，但這在 UTC 後端容器跑會把字串再次解析成本地時區時間，產生雙重轉換錯誤。

修法：改用 `Intl.DateTimeFormat` 顯式取年月日：

```js
const fmt = new Intl.DateTimeFormat('en-CA', {
  timeZone: 'Asia/Taipei',
  year: 'numeric', month: '2-digit', day: '2-digit',
});
const tomorrowTaipei = fmt.format(new Date(Date.now() + 24 * 3600 * 1000));
```

cron 排程器本身用 `node-cron` 的 `timezone: 'Asia/Taipei'` 選項，無論後端容器 TZ 是什麼都跑台北時間。

## 上線流程

### prod 環境變數核對

`m_b_每日行程推播_backend` 程式碼依賴的環境變數，prod 必須齊備（dev 後端已驗證會 work，照抄即可）：

| 變數 | 必需 | 說明 |
|---|---|---|
| `LINE_CHANNEL_ACCESS_TOKEN` | ✅ | 已存在於 prod（OAuth + push 共用 channel token） |
| `LINE_CHANNEL_SECRET` | ✅ | 已存在 |
| `DATABASE_URL` | ✅ | 已存在（內網 `postgresql.zeabur.internal:5432`） |
| `FRONTEND_URL` | ✅ | 已存在（用於 Flex 字卡按鈕指回前端） |
| `CRON_SECRET` | 暫不需要 | 目前 cron 用 node-cron 內部排程，未走外部 trigger，理論上不需要。但 README 已列為必填欄位（保留為日後外部 trigger 使用） |
| `TZ` | ⚠️ 不必要 | scheduler 自帶 `timezone: 'Asia/Taipei'`，容器 TZ 可保持 UTC 不影響 |

### 預期上線後行為

- prod 後端啟動 → 自動建 `system_settings` 表（idempotent，已存在則不動）+ 三筆預設值
- 預設 `time=21:00 / enabled=true / target=developer` → **第一晚 prod 會在 21:00 推播給開發者一人**
- 不是預期的 23:30 全社團推播（dev DB 內這三筆值是手動改過的）
- 想要與 dev 行為對齊：上線後 PC 連 prod DB 公網（暫開）→ `UPDATE system_settings SET value='23:30' WHERE key='daily_agenda_time'; UPDATE system_settings SET value='all' WHERE key='daily_agenda_target';` → 重啟 prod 後端讓 scheduler 重排
- 或等前端 6.x 上線後從 UI 改

### 雙份推播風險

- **dev 後端的 scheduler 也還在跑** — 但 v2.1.0 後 dev DB 為空（除使用者自己測試 onboarding 留下的記錄），實質名單接近 0，不會雙推
- 但 dev 後端如果未來補測試資料 + 開發者 LINE userId 對齊真人 → 會雙推。**保險做法**：上線後在 dev DB 把 `daily_agenda_enabled` 改為 `false` 停掉 dev 排程

## 副作用 / 後續注意

- **首晚推播時間預期 21:00 不是 23:30**：要對齊 dev 行為需在上線後手動改 prod DB
- **首晚推播對象預期 developer 一人**：要全社團需改 `daily_agenda_target=all`
- **前端設定頁未做**：所有調整目前要靠 PC 連 prod DB 改 system_settings 表 / 用 curl 打 PUT API 帶 bearer token
- **`m_b_每日行程推播_frontend` 還沒同步 main**：上線後要先把 main merge 進 frontend 分支（拿 v2.2.0 的後端 code 做 6.x ~ 7.x 開發）

## 教訓

- **dev 連 prod DB 是雙刃劍**：v2.1.0 之前 dev 後端共用 prod DB，意外讓推播在「prod 真實成員 × dev 後端 code」組合下提前驗證了好幾天，等於免費灰度測試。但這也是 v2.1.0 物理隔離前的危險狀態（dev 任何 bug 都會直接寫進 prod DB），不是好設計，只是運氣好沒出事
- **預設值與實際值落差**：dev 上 system_settings 被手動調過（21:00 → 23:30、developer → all），但 main 上線跑的是 server.js 內預設值。文件必須明確標出「預設 vs 期望狀態」，避免上線後第一晚行為與 dev 不一致時被當成 bug
