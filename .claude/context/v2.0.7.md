# v2.0.7 — 新用戶 onboarding 強制流程修補

**Tag**: v2.0.7
**Date**: 2026-04-25
**Type**: hotfix（設計邏輯壞掉的修復）
**Branch flow**: main → hotfix/onboarding-completion-guard → main

## 背景

OpenSpec change 10 完成 dev DB 物理隔離後，連續修了兩個 hotfix（v2.0.5 / v2.0.6）讓新用戶可以從 LINE 登入走到 `/profile/edit` 編輯頁。但測試中發現：**新用戶設計上應該照「用戶資料 → 用戶數據 → 主應用」順序完成**，現有程式碼完全沒有這層 guard，新用戶可以隨時跳過數據編輯直接進主應用，業務邏輯被打破。

從原設計者角度這是 bug — 邏輯壞掉。按工作流規則「main 上的 bug 開 hotfix」處理。

## 業務規則

| 階段 | 條件 | 行為 |
|---|---|---|
| 1. 未完成用戶資料 | members 表 `realName` / `email` / `phone` / `birthday` 任一為空 | 強制 `/profile/edit`，其他頁面全部不准進 |
| 2. 用戶資料完成但用戶數據未完成 | profile 完整 + `courseRecord` 為空（無任何課程紀錄） | 強制 `/user-stats/edit`，但允許回頭改 `/profile/edit` |
| 3. 兩者皆完成 | 上述兩個條件都滿足 | 自由使用所有頁面 |

## 修法（4 個檔案）

### 1. `frontend/src/contexts/AuthContext.jsx`

新增兩個 helper，傳給 ProtectedRoute 與編輯頁使用：

```js
const isProfileComplete = useCallback((u) => {
  if (!u) return false
  return !!(u.realName?.trim() && u.email?.trim() && u.phone?.trim() && u.birthday)
}, [])

const isStatsComplete = useCallback((u) => {
  if (!u || !u.courseRecord) return false
  const courses = u.courseRecord.split(',').map(c => c.trim()).filter(c => c)
  return courses.length > 0
}, [])
```

### 2. `frontend/src/App.jsx`

`ProtectedRoute` 加二級判斷，用 `useLocation` 取當前路徑：

```js
if (!profileComplete && path !== '/profile/edit') {
  return <Navigate to="/profile/edit" replace />
}
if (profileComplete && !statsComplete && path !== '/user-stats/edit' && path !== '/profile/edit') {
  return <Navigate to="/user-stats/edit" replace />
}
```

允許 `/profile/edit` 留在「資料完成但數據未完成」階段是為了讓使用者可以回頭修改，不會被鎖死。

### 3. `frontend/src/pages/ProfileEdit.jsx`

- `handleConfirm` 把原本只檢查 `realName` 的 silent return 改成四欄逐一 alert：

```js
if (!form.realName?.trim()) { alert('請輸入真實姓名'); return }
if (!form.email?.trim()) { alert('請輸入 Email'); return }
if (!form.phone?.trim()) { alert('請輸入電話號碼'); return }
if (!form.birthday) { alert('請選擇生日'); return }
```

- `fields` 陣列四欄全加 `required: true`
- 新用戶完成資料後 `navigate('/user-stats/edit')`（取代原本 `'/'`），順勢進 onboarding 第二階段
- 頁頂條件渲染 banner：「⚠️ 新用戶請完成所有欄位...」

### 4. `frontend/src/pages/UserStatsEdit.jsx`

- `handleConfirm` 加課程紀錄至少 1 筆驗證：

```js
if (form.courseList.length === 0) { alert('請至少勾選一個課程紀錄'); return }
```

- 「課程紀錄」label 加紅星 `*` 必填標示
- 頁頂條件渲染 banner：「⚠️ 新用戶請至少勾選一個課程紀錄...」

## 驗證

新用戶 LINE OAuth 登入流程：

1. ✅ 登入後到 `/login` 看到「建立資料」按鈕
2. ✅ 點按鈕 → 進入 `/profile/edit`，頁頂顯示橘色 banner 提示
3. ✅ 嘗試手動改網址到 `/`、`/calendar`、`/members` 等 → 全部被 ProtectedRoute 強制導回 `/profile/edit`
4. ✅ 在 ProfileEdit 漏填任一欄按「確認/儲存」→ alert 提示哪欄漏填
5. ✅ 四欄填完按「確認/儲存」→ 自動 navigate 到 `/user-stats/edit`，頁頂顯示橘色 banner
6. ✅ 在 UserStatsEdit 不勾任何課程按「確認/儲存」→ alert「請至少勾選一個課程紀錄」
7. ✅ 勾至少 1 個課程按「確認/儲存」→ navigate 到 `/user-stats`，可自由進其他頁面

既有用戶（profile + stats 都完整）：

- ✅ 登入後直接進主應用，不會被 onboarding guard 攔截
- ✅ 編輯資料/數據時 banner 不顯示（onboarding flag = false）

## 關聯

- 接續 v2.0.5 / v2.0.6 的「dev DB 為空首次登入」邊界 case 修復
- 完整解決 OpenSpec change 10 暴露的「新用戶完整旅程」議題
- ProfileEdit 與 UserStatsEdit 的 onboarding banner 是視覺安全網，主要 enforce 仍由 ProtectedRoute 把守

## 教訓

設計時的「應該如此」邏輯（業務規則）跟程式碼實作的「實際如此」可能脫鉤多時 — 因為 prod 環境的歷史資料保證使用者已完成 onboarding，這層 guard 從未被「真實場景測試」過。

dev 環境物理隔離（change 10）強迫新人 cold-start 跑完整流程，把這類沉睡多時的設計-實作落差全部翻出來。這是 dev 隔離的副作用價值，不只防汙染。
