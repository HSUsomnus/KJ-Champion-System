# v1.5.1 上下文記憶

git tag: v1.5.1
日期：2026-03-14

---

## 做了什麼

### 修復登入後 overlay 卡住問題（主要 bug fix）

- **根本原因**：`initAuth()` 的 try-catch 範圍太大，`localStorage.setItem()` 或 `history.replaceState()` 若拋出例外，整個 auth 流程跳到 catch block，`authInitialized = false`，liffReady 以 `null userId` 發送，login overlay 永遠不消失
- **修正（`public/js/auth.js`）**：
  - `localStorage.setItem()` × 3 改為各自用獨立 try-catch 包住，失敗只 warn，不中斷主流程
  - `history.replaceState()` 改為獨立 try-catch，失敗時 URL 不清除但登入仍繼續
  - `currentUserId` 在 try-catch 之前就已從 URL params 設定，保證 liffReady 能正確帶到 userId

### 修復 profile.html 也卡在登入 overlay 的問題

- **問題**：entryGate 在 `isRegistered: false` 時導向 `/profile.html`（不帶 userId），若 localStorage 寫入失敗，profile.html 取不到 userId → auth.js 呼叫 `showLoginOverlay()`
- **修正（`public/index.html`）**：導向改為 `/profile.html?userId=xxx&auth=1`，auth.js 在 profile.html 也能從 URL 直接讀取 userId

### OAuth state URL 編碼修正

- **問題**：base64 state 字串含 `+`、`/`，未 URL 編碼放入 OAuth URL，Express `req.query` 的 `qs` 在某些情境可能錯誤解析
- **修正（`server/routes/auth.js`）**：`state=${encodeURIComponent(encodedState)}`

### profile.js 防重入修正

- **問題**：`initProfile()` 同時用 `isInitialized()` 即時呼叫和 `liffReady` 事件觸發，造成 `checkRegistration()` 執行兩次（雙重 API call）
- **修正（`public/js/profile.js`）**：加 `registrationChecked` flag，用 `runCheck()` 包住，確保只執行一次

### SW 快取升級

- `public/sw.js`：`CACHE_NAME` 從 `kj-cache-v4` 升至 `kj-cache-v5`，強制所有裝置清除舊快取並取得修復後的靜態資源

---

## 為什麼改

- 手機（LINE App 內建瀏覽器、Samsung 瀏覽器）及電腦皆無法完成登入：OAuth 成功後仍卡在要求登入頁面
- 根本原因是 localStorage/history.replaceState 在某些行動瀏覽器中可能拋出例外，導致整個 auth 流程中斷

---

## 踩過的坑

1. `initAuth()` 的大型 try-catch 範圍包含了非關鍵操作（localStorage、history.replaceState），導致這些次要失敗意外終止整個 auth 流程
2. entryGate 導向 profile.html 時沒帶 userId，過度依賴 localStorage 的可靠性
3. profile.js 的 `isInitialized()` 即時呼叫和 liffReady 事件都觸發 `checkRegistration()`，造成 race condition

---

## 未來維護注意

- SW 快取目前為 `kj-cache-v5`，下次修改靜態資源後升至 v6
- `initAuth()` 的 localStorage 操作都有獨立 try-catch，即使失敗也不影響登入
- profile.html 現在透過 URL params（`?userId=xxx&auth=1`）接收來自 entryGate 的 userId，更可靠
