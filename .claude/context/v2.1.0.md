# v2.1.0 — Zeabur 專案分離 + dev/prod 物理隔離

**Tag**: v2.1.0
**Date**: 2026-04-25
**Type**: feature（基礎建設 — OpenSpec change 10）
**Branch flow**: main → m_b_zeabur_projects_split → main

## 背景

OpenSpec change 08 試圖讓 dev 後端連獨立的 `postgresql-test` DB（同 Zeabur 專案內），但發現「同專案內網平面網路」的天然限制 — 跨服務內網互通，dev 後端理論上能透過 `postgresql.zeabur.internal` 連到 prod DB（任何 dev 環境變數誤填 / 程式碼 bug 都可能寫入 prod）。

→ 08 做到一半被 10 取代，10 採用「不同 Zeabur 專案物理隔離」徹底解。

## 目標架構

```
Zeabur project: kj-champion（正式環境，現有）
├── postgresql                    ← prod DB（公網【關】、僅內網）
└── kj-champion-system            ← prod 後端（main branch）
                                    DATABASE_URL → postgresql.zeabur.internal:5432

Zeabur project: kj-champion-dev（測試環境，新建）
├── postgresql-dev                ← test DB（公網【開】供 PC 維護）
└── kj-champion-system-dev        ← dev 後端（dev branch）
                                    DATABASE_URL → postgresql.zeabur.internal:5432
                                    （跨專案內網不通，跟 prod DB 物理隔離）
```

## 副作用 / 後續注意

- **PC 本機 `.env` 的 `DATABASE_URL`（prod 公網）日常無效**：v2.1.0 起 prod DB 公網關閉。要做 prod DB 維護（schema dump 等）必須先去 Zeabur Dashboard `kj-champion` → `postgresql` → 網路 → 連線埠轉送 toggle 暫時開
- **dev DB 公網密碼曾在 chat 暴露**：未來建議找時機輪替 dev DB 密碼（風險低於 prod，因 dev DB 只有假資料 / 結構，但仍應做）
- **新用戶在 dev 站登入會走完整 onboarding**：因 dev DB 為空，要測既有用戶行為要先在 dev DB 寫入完整 member 記錄

## 教訓

dev 環境的「物理隔離」不是「為了好聽」的安全強化 — 它是「強迫業務邏輯回歸測試」的副作用價值。共用 prod DB 的 dev 環境永遠走 happy path，無法觸發 cold-start 邊界 case；隔離後新 DB 起空表，所有沉睡多時的「設計層理應如此但程式碼沒做」的 bug 一次浮現。

下次再做基礎建設變更時應預期：物理隔離後**緊接著有一波 hotfix**，不是 bug，是業務邏輯回歸測試的副作用。
