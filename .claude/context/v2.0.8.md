# v2.0.8 — UserStatsEdit 完成 onboarding 後導向主頁

**Tag**: v2.0.8
**Date**: 2026-04-25
**Type**: hotfix（接續 v2.0.7 修補）
**Branch flow**: main → hotfix/onboarding-completion-redirect-home → main

## 背景

v2.0.7 onboarding 強制流程上線後，使用者實測完整新用戶流程：

1. ✅ LINE 登入 → /profile/edit（鎖定不放行其他頁）
2. ✅ 填完四欄 → 自動跳 /user-stats/edit（鎖定）
3. ✅ 嘗試手動跳其他頁被擋回
4. ✅ 課程不勾按確認 → alert 提示
5. ❌ 勾課程按確認 → **跑去 `/user-stats` 數據頁**，但預期應該進主應用 `/`

## 根因

v2.0.7 寫死 `setTimeout(() => navigate('/user-stats'), 0)`，沒區分「新用戶第一次完成 onboarding」與「既有用戶回頭編輯數據」兩種情境。

兩種情境的合理導向不同：

| 情境 | 進入 `/user-stats/edit` 的來源 | 完成後預期導向 |
|---|---|---|
| 新用戶 onboarding 最後一步 | ProfileEdit 自動 navigate 過來 | `/`（進入主應用） |
| 既有用戶編輯數據 | /user-stats 點「編輯數據」 | `/user-stats`（回去看自己的數據） |

## 修法

UserStatsEdit 已有 `onboarding` 變數（`const onboarding = !isStatsComplete(user)`），直接用三元式：

```js
setTimeout(() => navigate(onboarding ? '/' : '/user-stats'), 0)
```

handleConfirm 的 closure 抓到的 `onboarding` 是「點按鈕當下」的值 — 此時 `isStatsComplete` 仍為 false（因為 user.courseRecord 還沒被 refreshUser 更新），所以 `onboarding=true` 判斷正確。
