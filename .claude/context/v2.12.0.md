# Context v2.12.0 — 模型分層工作證

## 背景

change 23 實現「模型分層工作證」三層角色邊界系統。目標是確保：
- **規劃者**（Opus/Fable）提出方案後無法意外寫入產品程式碼
- **實作者**（Sonnet）無法直接 push main、執行 git tag，必須經過收尾員流程
- **收尾員**（Haiku）自動化功能上線收尾（CHANGELOG / context / now.md / tag 指令）

此設計強化角色邊界、消除人工操作漏洞、規範化上線流程。

## 關鍵設計決策

### 1. Role 標記存檔案（`.claude/.session-role`）而非環境變數

- **選擇**：寫入檔案系統
- **理由**：
  - 環境變數在子代理/Agent 邊界會遺失（每個新 agent 環境初始化時不繼承上層 session 的環境變數）
  - 檔案系統 session 私有（worktree 隔離、`.gitignore` 排除，不污染 git）
  - hook 讀檔案系統可靠性高於嘗試環境變數傳遞

### 2. Role guard 不攔截 commit 執行本身，只攔截特定指令

- **選擇**：不攔截 `git commit`，改攔截 `git tag` 與 `git push origin main`
- **理由**：
  - engineer 仍需執行 commit（流程需要），只是禁止上線操作
  - commit 內容由 git-guard.js 的「產品碼 deny」規則把關（跨角色通用）
  - engineer 執行 commit 後→收尾員 agent 才負責 tag + push 指令準備

### 3. 角色無標記時完全不影響（向下相容）

- **選擇**：role-guard.js 先檢查 `.claude/.session-role` 是否存在，不存在就返回 `permissionDecision: none`
- **理由**：
  - 已有的 session（未打卡）應運行如常
  - 避免迴圈流程中冷啟動被意外 deny

### 4. 收尾員不自行 push main，只產出指令給使用者確認

- **選擇**：收尾員在 Haiku agent 階段產出 tag + push 指令文本，不執行
- **理由**：
  - main 變更涉及實際用戶，最高確認要求應來自使用者本人
  - CCR 沙箱環境 tag push 會 403，需使用者補做，與其預先自動失敗，不如統一流程
  - 分工清晰：AI（收尾員）負責記錄編製、使用者負責上線權限點擊

## 學習日誌

### 問題 1：新指令檔案沒被 git 追蹤到

**遇到的問題**：Phase 1 建立 `.claude/commands/規劃.md` 與 `實作.md` 後，`git status` 完全沒顯示這兩個新檔案，即使 `.gitignore` 已有 `!.claude/commands/*.md` 例外規則。

**試過的方法**：
- 先確認檔案確實存在 → ✅ 存在
- 確認例外規則寫法看起來正確 → ✅ 寫法正確
- 用 `git check-ignore -v` 直接檢查 → ❌ 發現規則被 `.gitignore:28:.claude/commands/` 這行擋住

**最終解法**：把 `.gitignore` 的 `.claude/commands/`（目錄整體排除）改成 `.claude/commands/*`（排除目錄內容，不排除目錄本身），例外規則才生效。

**根本原因**：git 的規則是「父目錄本身被排除時，無法用後面的例外規則重新納入其中的檔案」——舊寫法排除的是目錄本身，`!.claude/commands/*.md` 對新檔案不生效；既有的 `子代理.md` 能被追蹤到，是因為它是規則加入前就已經 commit 過的既有追蹤檔案，不受此限制。

## 後續注意

1. **第一個新 change 驗收流程實地檢查**：change 24 開發時須使用 `/規劃` → `/實作` → 收尾員完整流程，確認角色邊界與權限 deny 機制在實戰中正常運作。若發現 deny 時機不對或訊息不清，需回報並改進 role-guard.js。
