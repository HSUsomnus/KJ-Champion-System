## 關鍵設計決策

1. **Skill 化而非精簡規則文字**：常駐 token 降本的主力不是刪減規則內容，而是把規則搬到「依情境自動載入」的 Claude Code Skill 機制，讓大部分規則只在真正需要時才進上下文。
2. **hook 硬攔截取代 prose 勸告**：鐵律（禁止 main 直接 commit 產品碼、禁止 `git add -A`）改用 `permissionDecision: deny`，讓規則在動作發生的當下真正擋下，而非事後靠 Claude 自律。
3. **sync-branches.sh 衝突處理不自動覆蓋**：原 spec 設計是衝突時自動採 `-X theirs`，執行過程中經使用者明確要求改為預設停下回報，`SYNC_STRATEGY=theirs` 作為使用者已確認衝突內容後的手動開關——避免自動化靜默覆蓋分支內容的風險。
4. **README 全分支重寫規則降級**：原規則「不論哪個分支、push 前都要重寫 README」過度嚴格且與其他規則矛盾，改為「僅『功能上線』merge main 時完整重寫」。

## 學習日誌

### 問題 1：commit 型別前綴 deny 閘對 heredoc 寫法全面誤判

**遇到的問題**：補強工作證（addendum）新增的 commit message 型別前綴 deny 閘，正規表達式 `-m\s+["']([^"']+)` 只讀 shell 展開**前**的原始指令字串。Claude Code 標準推薦的 heredoc commit 寫法 `-m "$(cat <<'EOF' ... EOF)"` 在展開前開頭是 `$(cat <<`，被誤判成沒有型別前綴，整個 Bash 呼叫遭拒（不是訊息不合格，是指令完全沒執行）。
**試過的方法**：一度改用單行 commit message 繞過，但這只是治標不治本，該閘對之後所有 heredoc commit 都會持續誤判。
**最終解法**：訊息擷取邏輯改三層判定：單行 `-m` 直接驗前綴；heredoc 寫法解析 `<<MARKER ... MARKER` 區塊取內文第一行驗前綴；其他動態訊息（如 `$(date)`）靜態驗不了則 fail-open 放行，不硬擋。8 個測試案例（單行/heredoc × 合法/非法前綴、雙引號 marker、scope 前綴、內文提及 git 指令的誤判防護、動態訊息 fail-open）+ 3 項既有 regression 全數通過。
**根本原因**：hook 讀到的是 PreToolUse 時點 Claude 即將執行的**原始 Bash 指令字串**，此時 shell 尚未展開任何 `$(...)`、heredoc 等語法；任何以正規表達式驗證 commit 訊息內容的 hook，都必須考慮到這一層，不能假設收到的字串已是最終訊息。
